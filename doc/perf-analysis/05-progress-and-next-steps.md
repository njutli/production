## 三、回答关键疑问

### Q1：10% 提升是波动还是真实提升？

**判断：是真实提升，不是纯波动。** 依据有二：

1. **吞吐数字**：80.8 → 88.9 MiB/s（+10%）。单看这一个数，确实可能落在
   测试波动范围内（rados bench 都有 Stddev 25 的抖动），不足以单独定论。
2. **但 node3 网卡峰值 61% → 86% 是结构性证据**：node3 从"纯 OSD 节点"
   变成了"也承担 RGW 入口+扇出"，网卡负载明显上移。这说明 **LB 确实把
   一部分入口流量分到了 node3**，流量路径变了——这不是波动能解释的。

> 结论：**LB 已生效，提升真实**。只是幅度被另一个因素压住了（见 Q2/Q4）。

### Q2：要不要解决 node2 空间后再加第 3 个 RGW？

**先别急着上第 3 个 RGW，原因是"收益空间已经很小"。**

- 双 RGW 端到端 88.9，而后端裸能力 ~108 —— **离后端天花板只剩 ~19MB/s**。
- 加第 3 个 RGW 最多把端到端推到接近 108，**理论上限提升空间 < 20%**，
  且要先付出"给 node2 扩容/重分区"的代价（node2 根盘仅 20G，有风险）。
- 更关键：双 RGW 下 **node1 网卡仍是 100%**，说明流量分得**不均**
  （node1 100% vs node3 86%）。**当务之急不是加第 3 个，而是先把已有
  两个 RGW 的负载均衡调匀**——若两个都没用满就加第三个，是浪费。

> 建议顺序：① 先查 LB 实际分流比例、调匀 node1/node3（见后续方向 A）；
> ② 若调匀后端到端逼近 108 仍不够用，**才**考虑解决 node2 空间上第 3 个；
> ③ 但要清醒：第 3 个 RGW 的天花板还是后端 ~108，提升有限。

### Q3：当前瓶颈是不是已经变成"后端裸能力"了？

**部分是，但还不完全是——现在是"后端能力"和"RGW 入口不均"两者叠加。**

- 端到端 88.9 已接近后端 108，说明瓶颈**正在从"单 RGW 入口"转移到
  "后端 EC+HDD 整体能力"**。
- 但 node1 网卡 100%（node3 才 86%）表明 **RGW 入口这一层还没完全让开**
  ——流量没分匀，node1 仍是局部瓶颈。
- 所以现在是**过渡态**：入口瓶颈缓解了大半，后端瓶颈开始显现，两者并存。

### Q4：为何端到端还不到后端裸能力的 90%（88.9/108 ≈ 82%）？能否再提升？

差这 ~18% 主要来自**多出来的一层 RGW/S3 软件栈开销 + 流量不均**：

1. **RGW/S3 协议栈开销**：端到端要多走
   `JuiceFS(FUSE) → HTTP/S3 → RGW 解析 → 再写 RADOS`，
   而 rados bench 直接 `librados → OSD`。多出的 HTTP 封装、RGW 转发、
   FUSE 用户态拷贝，天然吃掉一截（~10~15% 很常见）。
2. **LB 转发 + 流量不均**：流量先到 tikv-node 的 HAProxy 再转发，多一跳；
   且 node1/node3 分得不匀（100% vs 86%），node1 先到顶就拖住整体。
3. **rados bench 自身也不是绝对上限**：它从 node2 单点发起，那条千兆和
   EC 扇出同样有约束，108 本身也偏保守。

**能否再提升：有限的空间。** 大致天花板就在 ~100~110MB/s：
- 把 LB 调匀（node1 不再 100%）→ 端到端有望从 88.9 提到 ~95~100；
- 但**再往上撞的就是后端 EC 4+2 + HDD 的硬上限**，软件层调不动了。

## 四、后续瓶颈分析与调优方向（按优先级 + 收益）

### 方向 A（先做，低成本）：调匀两个 RGW 的负载

- **现象**：双 RGW 下 node1 网卡 100%、node3 才 86%，分流不均。
- **查证**：看 HAProxy 后端实际分流比例
  ```bash
  echo "show stat" | sudo socat - /run/haproxy/admin.sock | \
    awk -F, '/rgw/{print $1,$2,"req:",$8,"bytes:",$9}'
  ```
- **可能原因**：JuiceFS 对 S3 是长连接，`balance roundrobin` 按连接轮询，
  少量长连接会分不匀；且 node1 既是 RGW 又是 OSD，本就更重。
- **调优**：① LB 改 `balance leastconn`（对长连接更均）；
  ② 提高 JuiceFS S3 并发（`--max-uploads` 等）让连接数更多、更易均衡。
- **预期**：端到端 88.9 → ~95~100MB/s。**先榨干已有两个 RGW，再谈第三个。**

### 方向 B（验证后端是否真到顶）：把 rados bench 做扎实

- 现有 108 是 30s 单点（node2）发起、样本偏少。
- **改进**：跑 60s，并**从多个节点同时发起** rados bench，看后端聚合
  能否超过 108（如果能，说明 108 是"单客户端单链路"限制而非后端绝对上限，
  那加 RGW 还有空间）。
- **同时三节点 node-monitor**：看后端被压满时，到底是**盘 util 到 100%**
  （真到 HDD 上限）还是**网卡先满**（还是网络限制）。这决定下一步该动盘还是网络。

### 方向 C（确认后才投入）：解决 node2 空间 + 第 3 个 RGW

- **前提**：方向 A 调匀后两个 RGW 都用满、且方向 B 证明后端还有余量。
- node2 根盘仅 20G（剩 ~2G），需先扩容/重分区（sda 有 ~426G 未分配）。
- **预期**：端到端推向 ~108 后端上限，提升空间 < 20%。

### 方向 D（针对随机负载，单列）：128 并发随机仍只有 36MB/s

- 顺序吞吐已接近后端上限，但**128 并发随机写仅 36MB/s**，差距巨大。
- 这是 **HDD + EC 4+2 随机小写读改写放大**导致，**和 RGW/LB 无关**。
- 加 RGW 救不了随机性能。要提升随机负载，方向是**后端介质升级
  （SSD / NVMe）**——但受 EC 4+2 硬规格约束，且属于硬件投入，非软件调优。

### 明确的天花板认知

- **顺序吞吐**：软件层（多 RGW+LB+调均衡）天花板 ~100~110MB/s（后端 EC+HDD）。
- **随机吞吐**：受 HDD 随机 IO 限制，~36MB/s，软件调不动，需换 SSD。
- 想突破这两个上限，只能动**后端硬件**（SSD、更多 OSD、更高网络），
  软件侧已接近榨干。

## 五、一句话进展总结

网络瓶颈已解决（百兆→千兆，吞吐 10×）；单 RGW 入口瓶颈已通过双 RGW+LB
**部分解决**（80→89MB/s，真实提升、非波动）；当前瓶颈处于**"RGW 入口
不均 + 后端 EC/HDD 能力"叠加的过渡态**。下一步先**调匀两个 RGW**（低成本、
预期到 ~95~100），并**做扎实 rados bench 确认后端真实上限**，再决定是否值得
为第 3 个 RGW 解决 node2 空间——但软件侧天花板就在 ~108，随机负载的硬限制
需后端介质升级才能突破。
