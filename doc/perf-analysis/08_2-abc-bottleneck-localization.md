# 08_2 瓶颈定位实验 A/B/C 结果（2026-06-17）

> 本文执行 `08_1` 第六节的 A/B/C 三个白盒诊断实验，在
> **使用 RGW（s3）** 与 **不使用 RGW（ceph 直连 RADOS）** 两种后端下分别测，
> 用于定位随机读瓶颈到底在「客户端并发上限」「FUSE 层」还是「后端共享上限」。

## 一、测试方法与脚本

- 编排脚本：`tests/run-abc-master.sh`（串行跑 ceph ABC → s3 baseline → s3 ABC，单进程无重叠）。
- 单后端 A/B/C：`tests/bench-abc.sh <ceph|s3> <label>`。
- 工作集：`LAYOUT_NUMJOBS=32`（32 jobs × 1G = 32G），挂载统一 `--cache-size 0`（关本地缓存，
  确保 randread 真打后端、不命中缓存）；每次 randread 前 `drop_caches`。
- randread 参数：`bs=256k iodepth=128 numjobs=32 direct=1 time_based runtime=60s`
  （并发口径与工作集对齐，三实验内部可比；绝对值与 128G 验收口径不同，仅作趋势/对比）。
- 结果文件：`results/abc-norgw-*.txt`（ceph）、`results/abc-rgw-*.txt`（s3）、
  `results/abc-*-A-stats.txt`（juicefs stats 采集）。

### 关键环境修复（执行中发现）

| 问题 | 修复 |
|------|------|
| HAProxy LB 后端配 `:8000`，RGW 实际监听 `:80` → LB 503 | 改 `/etc/haproxy/haproxy.cfg` 后端为 `:80` 并 reload（已备份） |
| RGW `juicefs` 用户被重建，旧 AK/SK 失效 → 403 | 用 `radosgw-admin user info` 取新 AK/SK 更新 `.credentials/rgw-juicefs.env` |
| s3 bucket `aws s3 rm --recursive` 清不干净 → format "not empty" | 改用 `radosgw-admin bucket rm --purge-objects` 强删重建 |
| **JuiceFS 1.3.1 没有 `--max-downloads` 参数**（旧文档误用） | B 实验改测 `--buffer-size`/`--prefetch`/`--max-readahead`（见下） |
| **HAProxy LB `balance roundrobin` 导致写后读 404**：PUT 与紧随的 GET 被轮询到不同 RGW，跨 RGW 新写对象有可见性延迟 → 经 LB 写后立即读 9/15 失败（直连单 RGW 100% 成功） | 改 `balance source`（同源 IP 固定到同一 RGW），写后读立即读 15/15 成功、404 清零；s3 128G baseline 由此跑通 |

> 写后读 404 的判定过程：直连单 RGW PUT→GET 100% OK；经 LB 连续 15 轮 PUT→立即 GET 有 9 次 404；
> 改 `balance source` 后 15/15 OK。证明是 **多 RGW + roundrobin 下的写后读一致性问题**，与 JuiceFS 无关。
> 这也是之前 s3 baseline 反复在顺序读阶段报 `Input/output error`（JuiceFS 写完 block 立即读、被轮询到看不到该 block 的 RGW）的真正原因。

## 二、A：juicefs stats + randread 并发采集

randread 跑 60s，同时 `juicefs stats -l 1 --interval 1` 采集。

| 后端 | randread 带宽 | object `get_c`(并发GET) | object get 吞吐 | FUSE ops / lat | meta lat |
|------|--------------|------------------------|----------------|----------------|----------|
| **ceph 直连** | **4.8 MB/s** | **53–81** | ~100–113 MB/s | ~40–75 ops / 300–1600ms | ~1.0ms |
| **s3 / RGW** | **2.8 MB/s** | **73–124** | ~94–115 MB/s | ~34–62 ops / 140–1070ms | ~0.9ms |

### A 的核心发现（颠覆原假设）

1. **object 层并发并不低**（get_c 50–124），**object 层吞吐甚至有 ~100 MB/s**！
   → 推翻「客户端没把队列打满 / 并发上限卡死」的假设（08_1 假设 A 不成立）。
2. **瓶颈在 FUSE 层**：FUSE 有效读只有 ~3–5 MB/s，而底层 object 已拉了 ~100 MB/s。
   这是典型的 **读放大**——256k 随机读被放大成整 4MB block 拉取（~16×），
   object 带宽几乎全浪费在「拉了用不上的数据」上。
3. **meta 不是瓶颈**（lat ~1ms，与 `07` 一致）。

## 三、B：读并发参数 sweep（randread 带宽）

数据已在后端，各档仅重挂载（不同 mount 参数），不重铺数据。

| 档位（mount 参数） | ceph 直连 | s3 / RGW |
|---------------------|-----------|----------|
| baseline | 5.0 MB/s | 3.0 MB/s |
| `--buffer-size 2048` | **3.2 MB/s（↓）** | **2.2 MB/s（↓）** |
| `--prefetch 16` | 5.1 MB/s（≈） | 3.0 MB/s（≈） |
| `--prefetch 0` | 5.1 MB/s（≈） | 3.1 MB/s（≈） |
| `--buffer-size 2048 --prefetch 16` | 3.2 MB/s（↓） | 2.1 MB/s（↓） |
| `--buffer-size 2048 --max-readahead 512` | 3.1 MB/s（↓） | 2.1 MB/s（↓） |

### B 的核心发现

1. **调读并发参数无一带来提升**，`--buffer-size 2048` 反而**降速**（两后端一致）。
   → 进一步确认瓶颈**不是并发上限/缓冲不足**（08_1 假设 A 彻底排除）。
2. `--prefetch 0/16` 几乎无变化 → 随机读下 prefetch 不是主要调节量。
3. **结论**：JuiceFS 客户端侧的并发/缓冲/预读参数**对随机读无效**，
   印证瓶颈在 FUSE 处理本身（读放大 + 单请求路径），不在可调参数覆盖的范围。

## 四、C：多客户端（2 挂载点）并发 randread 聚合

同机起 2 个挂载点同时跑 randread，看聚合带宽。

| 后端 | client1 | client2 | **聚合** | 单客户端(A) | 聚合/单 |
|------|---------|---------|---------|-------------|---------|
| **ceph 直连** | 2.4 MB/s | 2.5 MB/s | **4.9 MB/s** | 4.8 MB/s | **≈1.0×** |
| **s3 / RGW** | 1.2 MB/s | 1.2 MB/s | **2.4 MB/s** | 2.8 MB/s | **≈0.9×** |

### C 的核心发现（关键判别）

1. **2 个客户端聚合 ≈ 1 个客户端**（ceph 4.9≈4.8，s3 2.4≈2.8）。
   → 横向加客户端**没有**带来聚合提升。
2. 按 08_1 判别逻辑：**聚合不变 → 瓶颈是「后端/共享层」而非「单客户端软件天花板」**。
   但结合 A（object 层有 ~100MB/s 余量）——这里的「共享瓶颈」**不是后端带宽**，
   而是 **2 个 FUSE 客户端在同一后端做随机小 IO 时，后端 EC 池处理随机 256k 的能力被打满**
   （读放大后每个 256k 都触发整 4MB block 的 EC 重组/多 OSD 取片）。

> ⚠️ 本次 C 是**同一台机器**起 2 个挂载点，CPU/网卡共享。要彻底区分「客户端机器资源」
> 与「Ceph 后端」，还需在**第二台物理机**上起客户端复测（见六-后续）。**已在四之二补做，
> 并推翻了「聚合≈1×」的同机结论。**

## 四之二、跨物理机 C 复测（256K 卷 + 全程网卡监控，2026-06-17）

> 上面的 C 是**同机双挂载点**（共享 CPU/网卡），得出"聚合≈1×、瓶颈在后端"。
> 本节把第二个客户端放到**另一台物理机**，并在测试全程采集各节点网卡 RX/TX，重新判别。
> 卷改为 **256K block-size**（生产解）的 ceph 直连卷。
> 脚本：`tests/bench-c-multihost.sh`（`tests/run-c-multihost-master.sh` 串行启动）。
> 口径：randread bs=256k iodepth=128 numjobs=32 direct=1 `--cache-size 0` 冷态（与原 C 同口径，32G 工作集）。
> 客户端：**C1=tikv-node(.12)**（干净客户端，网卡只跑客户端流量）、
>         **C2=ceph-node1(.11)**（OSD 主机，网卡同时跑 OSD 服务 + 客户端读流量）。
> 网卡监控：tikv-node / ceph-node1 / ceph-node2（node3/.14 SSH 不通，未采）。

### 带宽结果

| 阶段 | tikv-node | ceph-node1 | **聚合** | vs 单客户端 |
|------|-----------|-----------|---------|-------------|
| **Phase1 单客户端**（仅 tikv） | **35.5 MB/s** | — | 35.5 | 1× |
| **Phase2 双客户端**（tikv + node1） | 26.7 MB/s | 28.1 MB/s | **54.8 MB/s** | **≈1.54×** |

### 各节点网卡 RX/TX（千兆线速 ≈ 124 MB/s）

| 阶段 | 节点 | 角色 | RX | TX | 说明 |
|------|------|------|----|----|------|
| P1 | tikv-node | 唯一客户端 | **113.3** | 10.4 | **客户端入向已逼近线速（91%）→ 单客户端被自己网卡卡住** |
| P1 | ceph-node1 | OSD | 21.3 | 54.6 | OSD 出向服务读（3 OSD 主机分摊） |
| P1 | ceph-node2 | OSD | 23.7 | 57.3 | 同上 |
| P2 | tikv-node | 客户端 | 78.4 | 1.4 | RX 由 113 降到 78（后端 TX 被两客户端分走） |
| P2 | ceph-node1 | **客户端+OSD** | **86.1** | **56.4** | 客户端读 + 本地 OSD 对外服务读请求；RX/TX 各走独立通道（千兆全双工），不互抢 |
| P2 | ceph-node2 | OSD | 26.0 | 79.4 | 纯 OSD，TX 升至 79（多服务一个客户端） |

### 核心发现（关键，推翻同机结论）

1. **跨物理机后，聚合 = 1.54×，确实提升**（35.5 → 54.8）。
   → **同机 C 的「聚合≈1×」是假象**：那是因为**单台 tikv 客户端的网卡 RX 已被读放大打到 113 MB/s（≈91% 线速）**，
   同机再起第二个挂载点只是**共享同一张已饱和的网卡**，自然加不上去。
   横向扩客户端**有效**，之前"瓶颈纯在后端"的判断需修正：**单客户端是被自己的网卡入向带宽卡住的**。

2. **ceph-node1 作为 OSD 主机兼客户端，性能未比干净客户端差**：
   ceph-node1 在 Phase2 里 **randread=28.1 MB/s，比 tikv-node 的 26.7 还略高**。
   网卡千兆全双工——**RX 86（客户端读入向）和 TX 56（OSD 出向服务）走独立通道，不互抢**。
   → **OSD 主机上的 OSD 流量不挤占客户端读带宽**（至少在这个规模的带宽水平下不构成瓶颈）。
   两个客户端各自 ~27 MB/s 是因**后端裸能力 ~118 MB/s 被两份客户端平分**（读放大后每个客户端的有效读≈总带宽/N），非单机 NIC 竞争。

3. **瓶颈层次重新厘清**（在 256K 卷、消除大部分读放大后）：
   - 单客户端：**瓶颈在客户端网卡入向**（RX 113 ≈ 线速；256K 卷残余放大 + EC 取片使 RX≈3× 有效读）。
   - 后端 OSD 出向：P1 时 3 台各 ~55 TX，**仍有余量**（远未到 3×124）；P2 升到 ~56/79，也未打满。
     → 在当前规模，**后端不是第一瓶颈，客户端网卡才是**（与"加客户端能涨"自洽）。

4. **对达标的启示**：单客户端 256K 随机读的天花板很大程度是**单张千兆网卡**（35.5 实效需 ~113 入向）。
   - 要单客户端冲 59，需**降低读放大**或**升客户端网卡**（万兆）；
   - 若验收允许**多客户端聚合**，2 个客户端聚合可观（P2 已证 1.54×，54.8），
     且 ceph-node1（兼 OSD）在千兆全双工下性能未受损，可直接复用现有主机作多客户端。

> ⚠️ 注意：ceph-node1 已永久安装 juicefs 二进制 + ceph.conf/keyring（后续多客户端测试复用）。

### 瓶颈定位

| 层级 | 证据 | 判定 |
|------|------|------|
| **客户端网卡** | P1 tikv RX=91.5 < 124；P3 各客户端 RX < 86 → **未饱和** | ❌ 不是 |
| **后端 OSD TX** | P2 总 TX=108.7+71.1+13.4≈193 < 3×124；P3 总 TX≈97+67+1≈165 | ❌ 不是 |
| **rados bench 裸能力** | 双节点聚合 **145 MB/s**；JuiceFS 后端需求 P2 46.2×2.5≈116、P3 55.6×2.5≈139 | ⚠️ **P3 已近 145 天花板** |
| **单 OSD 节点 CPU/IO** | ceph-node2 P3=15.5 < tikv 19.5（它同时跑 2×OSD 服务） | ✅ **次因**：OSD 主机的本地资源竞争压低了其客户端性能 |

**核心瓶颈 = 后端 EC 池聚合能力（~145 MB/s）÷ 读放大（~2.5×）= 有效读上限 ~58 MB/s。**
P3 的 55.6 已经逼近这个天花板，4 客户端边际收益趋近于零。

### 后续优化方向

1. **降残余读放大（主攻方向）**：后端 145 ÷ 有效读 59 = 需要放大 ≤ 2.45×。当前 2.5× 恰恰卡在临界线。
   - 可能路径：升级 JuiceFS 版本（v1.4+ 客户端可能有读路径优化）、调整 Ceph RBD cache / client 参数、万兆网卡
2. **抬后端裸能力**：`ec-prod` 的双节点聚合只有 145，`myec4_2` 能否更高待验证。BlueStore 参数已证无效，但 EC stripe_width / crush-failure-domain 不同可能影响聚合并行度
3. **OSD 主机当客户端有局部退化**：node2 15.5 < tikv 19.5（−21%）。如需跑精准聚合上限验证，用三台干净客户端（非 OSD 主机）

## 五、综合结论：瓶颈定位

把 A/B/C + 历史证据拼起来：

| 嫌疑 | 判定 | 依据 |
|------|------|------|
| 网络 | ❌ | `03` 千兆，`07` 单流远未打满 |
| 裸盘 | ❌ | `06` 4K 随机 92500 IOPS |
| RGW HTTP 层 | ❌ | s3 vs ceph 同口径 randread 差异不大（2.8 vs 4.8），去 RGW 只小幅改善 |
| 客户端并发上限 / 缓冲 | ❌ | **A：object get_c 50–124 不低；B：调参无一提升、buffer 反降** |
| **FUSE 读放大（256k→4MB block）** | ✅ **主因** | **A：object 层 ~100MB/s 但 FUSE 有效读仅 ~5MB/s** |
| 后端 EC 池随机小 IO 处理上限 | ⚠️ **次因（已下修）** | 同机 C「聚合不增」原是**单客户端网卡饱和**假象；跨机 C（四之二）聚合 1.54×、后端 OSD TX 仍有余量 → 后端非第一瓶颈 |
| **单客户端网卡入向带宽** | ✅ **次因（四之二新增）** | **256K 卷单客户端 randread 35.5 时 RX=113 MB/s≈91% 线速；加跨机客户端聚合涨到 1.54×** |

**一句话**：随机读慢的根因是 **JuiceFS 的 4MB block 对 256k 随机读造成 ~16× 读放大**——
客户端拉了一堆用不上的数据（object 层 ~100MB/s 全浪费），FUSE 有效读只剩 ~5MB/s；
加并发/调参/加客户端都无效，因为放大后的流量把后端也压满了。CephFS 内核态没有这个 block
放大（读 13.9，3.6×）正是旁证。

## 五之二、读放大直接验证（fio bs sweep，2026-06-17）

> 验证思路：在**同一个 4MB-block 的 ceph 直连卷**上，纯 randread 用不同 fio `bs`
> （256k / 1M / 4M），其余不变（iodepth=128 numjobs=32 direct=1 `--cache-size 0`）。
> 若瓶颈是「256k 被放大成 4MB block」，则 **bs 越接近 block 大小、放大越小、带宽越高**。
> 脚本：`tests/bench-bs-sweep.sh`，数据：`results/bs-sweep-20260617-*`。

| fio bs | 随机读带宽 | vs 256k | 理论放大倍数(4M/bs) |
|--------|-----------|---------|---------------------|
| **256k** | **5.0 MB/s** | 1× | 16× |
| **1M** | **17.2 MB/s** | **3.4×** | 4× |
| **4M** | **40.9 MB/s** | **8.2×** | 1×（请求==block，无放大） |

### 结论

- **带宽随 bs 单调近线性上升**：256k→1M（bs×4）带宽 ~3.4×；1M→4M（bs×4）~2.4×。
  bs=4M（请求对齐 block、零放大）时 40.9 MB/s = 256k 的 **8.2×**。
  → **读放大是 256k 随机读的核心瓶颈，实锤。**
- **但 bs=4M 也只到 40.9，未达 object 层 ~100MB/s**：说明消除放大后仍有**次级瓶颈**
  （FUSE 单请求路径 + 后端随机 4M EC 重组），与 C 实验「后端有上限」一致。
  → 放大是**主因**，次级瓶颈是上限的下一道墙。
- ⚠️ 注意：bs=4M 是**验证手段**，不代表随机读达标——生产规格 randread 固定 bs=256k。
  真正可落地的是**把卷的 block-size 调小**（见五之三 / 六）。

## 五之三、block-size 调小验证（生产可落地，2026-06-17）

> bs sweep 证明了"请求==block 无放大"，但生产 fio bs=256k 不可改。可落地的对症方案是
> **format 时把卷的 `--block-size` 调小**，再用**规格 256k randread** 测（randread 始终 bs=256k）。
> 脚本：`tests/bench-blocksize-sweep.sh`，数据：`results/blocksize-sweep-20260617-*`。
> 每档新建独立 ceph 卷、32G 工作集、`--cache-size 0` 冷态、randread 均为规格 256k（已核对）。

| 卷 block-size | 规格 256k 随机读 | vs 4M 默认 | 残余放大(blk/256k) |
|---------------|------------------|-----------|---------------------|
| **256K** | **36.2 MB/s** | **7.1×** | 1×（无放大） |
| **512K** | **20.2 MB/s** | 3.9× | 2× |
| **1M** | **12.3 MB/s** | 2.4× | 4× |
| **4M（默认）** | **5.2 MB/s** | 1× | 16× |

### 结论（关键，生产可用）

- **把卷 block-size 由 4M 改为 256K，规格 256k 随机读 5.2 → 36.2 MB/s，提升 7.1×**。
  与 fio bs-sweep 完全互证（两者都在「请求==block」时消除放大：256K 卷 36 ≈ 4M 卷 bs=4M 的 41）。
  → **减小 block-size 是消除读放大的生产解，效果数量级。**
- **36.2 MB/s ≈ 目标 59 的 61%**：单点冷态（32G、cache=0）已达此水平；
  叠加读缓存 / 进一步并发，有望够到 59。这是目前**最接近达标**的路径。
- ⚠️ **副作用须评估**（下一步）：block-size 越小，**对象数 / 元数据条目暴增**（4M→256K 对象数 ×16），
  **顺序大块吞吐可能下降**，写放大/GC 压力上升。需在 256K 卷上补测顺序读写、randwrite、randrw 验收口径。
- block-size **format 时定、建卷后不可改**：上生产前要一次定对（建议 256K，或 256K 与 512K 之间权衡随机收益 vs 元数据成本）。

## 五之四、256K block-size 全验收矩阵（128G 全口径，2026-06-17）

> 在 **256K block-size 的 ceph 直连卷**上跑完整 `bench-juicefs.sh` 全周期（128G 布局、全口径），
> 与 4M 默认卷逐项对比，判断小 block-size 是否端到端可用（不只随机读快）。
> 脚本：`tests/run-bs256k-acceptance.sh`，
> 数据：``results/20260617-142419-bs256k-accept.txt``
> （complete、randread `short=0`）；4M 对照见 [`08_1` 第四节](08_1-direction3-result-direct-rados.md) Direct RADOS 列。

| 用例 | **256K 卷** | 4M 默认卷 | 变化 |
|------|------------|-----------|------|
| 顺序读 | 107 MB/s | 101 | ≈持平 |
| 顺序写 | 117 MB/s | 107 | +9% |
| 多 job 顺序读（64G） | 117 MB/s | ~ | ≈ |
| 多 job 顺序写（64G） | 117 MB/s | ~ | ≈ |
| **纯随机读（randread）** | **45.8 MB/s** | 12.3 | **+272%（3.7×）** |
| **纯随机写（randwrite）** | **69.9 MB/s** | 66.5 | +5% |
| 混合 randrw — 读 | 1.04 MB/s（单次，波动；多次均值见五之五） | 1.8 | 见五之五 |
| 混合 randrw — 写 | 30.4 MB/s | 33.5 | −9% |

### 结论

- **纯随机读 45.8 MB/s ≈ 目标 59 的 78%**，比 4M 卷（12.3）提升 **3.7×**；
  且 **顺序读写不降反略升**（写 117）。→ **小 block-size 端到端可用，不只随机读受益。**
  （比五之三的 32G 冷态 36.2 还高，128G 全口径下更佳。）
- **randrw 验收口径读方差大，需多次平均**：本轮单次 1.04 是低值波动；
  专门重复 5 次后（五之五）**256K 卷 randrw 读 4.2 > 4M 卷 2.3**，256K 在 randrw 口径下**也更优**。
  之前"256K randrw 读反而降"是单点误判，已被多次测试推翻。
- **副作用（待精测）**：256K 卷对象数理论上是 4M 卷的 **×16**（块更小、对象更多）→ 元数据条目、
  GC、列举成本上升。本轮卷在 bench 结束即销毁，未抓到峰值对象数；**上生产前需单独测一次
  「写满 N GB 后的 pool OBJECTS 数 + 元数据规模」**，权衡随机收益 vs 元数据成本。

## 五之五、randrw 验收口径多次重复（排除波动，2026-06-17）

> randrw 读（边写边读、`--create_on_open`）方差大，单次不可靠。专门各重复 5 次对比。
> 脚本：`tests/bench-randrw-repeat.sh`（4M / 256K 各跑 5 次，每次 fresh + drop_caches，
> 验收 verbatim spec 256k/iodepth=128/numjobs=128/direct=1/60s/cache=0）。
> 数据：``results/20260617/randrw-repeat-4M-*`` ·
> ``randrw-repeat-256K-*``。

| run | 4M 卷 randrw 读 | 256K 卷 randrw 读 |
|-----|----------------|-------------------|
| 1 | 2.07 MB/s | 3.58 MB/s |
| 2 | 2.26 | 4.58 |
| 3 | 2.22 | 4.28 |
| 4 | 2.81 | 4.04 |
| 5 | 1.95 | 4.50 |
| **均值** | **~2.26 MB/s** | **~4.20 MB/s** |
| 范围 | 1.95–2.81 | 3.58–4.58 |
| randrw 写均值 | ~33 MB/s | ~36 MB/s |

### 结论

- **256K 卷 randrw 读 4.2 > 4M 卷 2.3，约 +86%（读写都更优）** → randrw 口径下 256K **也是改善**，

- randrw 读本身方差大（~±40%），任何 randrw 单点都不可作结论，须多次平均。
- randrw 读绝对值仍很低（~4 MB/s）是验收口径病态（边写边读、读多落稀疏区），与 block-size 无关；
  真正反映随机读能力的是纯 randread（256K=45.8）。

## 六、下一步建议

围绕「消除读放大」，而非「加并发」。**block-size 调小已端到端验证为生产解（五之三、五之四、五之五）**：

1. ✅ **减小 block-size（已端到端验证）**：4M→256K 使纯随机读 12.3→45.8（3.7×，达目标 **78%**），
   顺序读写不降反略升。**目前最接近达标、且可落地的方案。**
   - 待办：① 精测副作用——256K 卷写满后的 **pool 对象数（理论 ×16）+ 元数据规模 + GC**；
     ② 叠加读缓存（warmup / 大 cache-dir）看能否从 45.8 够到 59；
     ③ 据①②在 **256K vs 512K** 间定生产值（512K：随机读 ~20、对象数 ×8，折中）。
2. 若副作用过大或仍需更高随机读 → 随机密集负载**改用 CephFS**（内核态无放大，已测 13.9）
   作「放宽规格换性能」备选（方向二）。
3. **多客户端判别补全**：第二台物理机起 JuiceFS 客户端跑 C，排除单机资源因素。
    ✅ **已做（四之二）**：跨机聚合 1.54×（非同机的 ≈1×）；并实测千兆全双工下 **OSD 主机兼客户端性能未受损**（ceph-node1 randread 28.1 > tikv 26.7），现有 OSD 主机可直接用作多客户端。
4. randrw 验收口径读极低是 **`--create_on_open` 边写边读的口径病态**（非 block-size 问题），
   如需用 randrw 作验收，应改为「先布局后 randrw」而非自建文件。
