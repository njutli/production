# 07 随机读写专项：目标、瓶颈定位与调优方向（2026-06-12）

> 阶段转向：顺序/rados bench 已确认触顶（软件栈天花板，非硬件，见 `06`/`06_1`/`06_2`），
> 现把重心转到**存储规格的验收口径——256k 随机读写**。
>
> **目标：随机读、写带宽各达到千兆网卡的 50% ≈ 59 MB/s。**

---

## 一、现状（256k randrw, iodepth=128, numjobs=128, direct=1）

| 方案 | 后端 | 随机读 | 随机写 | 距目标(59) |
|------|------|-------|-------|-----------|
| **JuiceFS** | SSD | **3.8** | **38** | 读差 ~15×，写接近 |
| JuiceFS | 内存盘 | 3.5 | 36.8 | 介质无关 |
| CephFS（内核态） | SSD | 13.9 | 13.8 | 读 3.6×↑，写↓ |
| CephFS | 内存盘 | 27.6 | 27.3 | 读写均衡，仍未达标 |

### 两个关键事实

1. **随机写 38 已接近目标 59**（差距小，靠并发/调参可能够到）；
   **随机读 3.8 是真正的拦路虎，差近一个数量级**——主攻方向。
2. **换内存盘对 JuiceFS 随机性能无影响**（3.8→3.5）→ 瓶颈**不在介质/后端**，
   在 **JuiceFS 的访问路径（FUSE + 元数据 TiKV + 对象 RGW 的多层往返）**。
   CephFS 同后端随机读立刻 3.8→13.9（3.6×）就是铁证：**瓶颈在 JuiceFS 框架本身**。

### 为什么 JuiceFS 随机读这么差

一次 256k 随机读，JuiceFS 要：
`FUSE 拦截 → 查 TiKV（chunk/slice 位置，一次网络往返） → 向 RGW GET 对象（HTTP，又一次往返）`。
每个随机 IO 都走这条全链路，**IOPS 被往返延迟钉死**，与介质无关。
而随机写较快（38）是因为 JuiceFS **写新 chunk 整块写**，绕过了 EC 的就地改写（RMW）。

---

## 二、根本判断

> **这套架构（JuiceFS → S3/RGW → EC）对随机小 IO 先天不友好**：每个随机 IO 都付出
> "FUSE + 元数据往返 + 对象往返"的固定延迟税。顺序大 IO 能摊薄这个税，随机小 IO 不能。
> 所以随机读优化的核心是**缩短/减少这条往返链路**，而不是再去碰已被排除的硬件/介质。

---

## 三、调优方向（按优先级，含框架内 + 换方案）

### A. JuiceFS 框架内（先做，低成本）

1. **调小 JuiceFS block-size（最可能见效，治随机读放大）**
   - JuiceFS 默认 block 4MB；256k 随机读却要拉取**整个 4MB block** → 读放大 ~16×。
   - 格式化时把 `--block-size` 调小（如 256k~1M），让随机读不必拉整块。
   - ⚠️ 需重新 format 文件系统才能改 block-size。

2. **量化定位随机读瓶颈（必做，避免盲目）**
   - `juicefs stats <mount>`：看随机读时是**元数据操作多**（TiKV 往返瓶颈）
     还是**对象 GET 多/大**（RGW + 读放大）。
   - 节点间 + 客户端到节点 **`ping` RTT**：随机 IOPS ≈ 1/RTT，RTT 高则随机被钉死。
     （这是"网络是瓶颈"在随机场景下**可能成立的维度——是延迟 RTT，不是带宽**。）
   - `iperf3`：确认带宽不是问题（顺序已证不是）。

3. **元数据引擎换 Redis**
   - TiKV 是分布式强一致 KV，往返多、延迟高；Redis 单点延迟低得多，
     JuiceFS 官方亦推荐 Redis 取最低元数据延迟。随机小 IO 对元数据延迟敏感。

4. **缓存策略**（取决于验收是否允许缓存）
   - `--cache-size` 调大、`--buffer-size`、`--attr-cache`/`--entry-cache`（元数据缓存）。
   - ⚠️ 验收用 `--direct=1` 会绕过部分缓存，**先确认验收口径是否允许缓存**，否则无效。

5. **多客户端 / 多挂载点并发**
   - 顺序场景多客户端聚合到 191 MB/s（单客户端 106）；随机也可能靠并发突破单流往返限制。

### B. 换方案（框架内榨干后认真评估）

| 方案 | 随机读 | 随机写 | 代价/约束 |
|------|-------|-------|----------|
| **CephFS（内核态）替代 JuiceFS** | 13.9（已测） | 13.8 | 无 FUSE 税；但 EC overwrite 触发 RMW 拖慢写；需 POSIX 共享语义 |
| **CephFS + 副本元数据/数据池** | 待测 | 待测 | 副本避开 EC RMW，写可能改善；副本违反 EC 4+2 规格 |
| **JuiceFS 直连 RADOS（去掉 RGW）** | 待测 | 待测 | 少一层 HTTP/RGW 往返，随机延迟可能降；仍有 FUSE+元数据 |
| **Ceph RBD + 本地文件系统** | 待测（通常远高） | 待测 | 块设备随机性能好；但单客户端独占，非多端共享 |

> 选型前提：取决于**业务访问模式**——是否必须多客户端共享 POSIX 文件？
> 若是 → CephFS/JuiceFS；若可块设备独占 → RBD 随机性能最佳。

### C. 已排除（不要在随机场景重复尝试）

- ❌ 换介质 / 更快盘（JuiceFS 随机换内存盘无效，3.8→3.5）
- ❌ 增大 OSD 数 / RAID 调参 / 调度器（顺序阶段已证无效，瓶颈不在后端硬件）

## 五、一句话

随机**写 38** 接近目标 59，随机**读 3.8** 差一个数量级是主战场。瓶颈不在硬件/介质
（换内存盘无效），而在 **JuiceFS 的 FUSE + 元数据 + RGW 多层往返**（CephFS 同后端读
即 3.6×）。优先：**①调小 block-size 治读放大 ②诊断 RTT/元数据定位 ③元数据换 Redis /
去 RGW 直连 RADOS ④评估 CephFS/RBD 换栈**。先诊断再动手，别碰已排除的硬件方向。
