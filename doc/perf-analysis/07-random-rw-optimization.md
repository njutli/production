# 07 随机读写专项：目标、瓶颈定位与调优方向（2026-06-12）

> 阶段转向：顺序/rados bench 已确认触顶（软件栈天花板，非硬件，见 `06`/`06_1`/`06_2`），
> 现把重心转到**存储规格的验收口径——256k 随机读写**。
>
> **目标：随机读、写带宽各达到千兆网卡的 50% ≈ 59 MB/s。**

---

## 一、现状（256k randrw, iodepth=128, numjobs=128, direct=1）

| 方案 | 后端 | 随机读 | 随机写 | 距目标(59) |
|------|------|-------|-------|-----------|
| **JuiceFS** | SSD | **3.8** | **38** | 读差 ~15×，写接近 |
| JuiceFS | 内存盘 | 3.5 | 36.8 | 介质无关 |
| CephFS（内核态） | SSD | 13.9 | 13.8 | 读 3.6×↑，写↓ |
| CephFS | 内存盘 | 27.6 | 27.3 | 读写均衡，仍未达标 |

### 两个关键事实

1. **随机写 38 已接近目标 59**（差距小，靠并发/调参可能够到）；
   **随机读 3.8 是真正的拦路虎，差近一个数量级**——主攻方向。
2. **换内存盘对 JuiceFS 随机性能无影响**（3.8→3.5）→ 瓶颈**不在介质/后端**，
   在 **JuiceFS 的访问路径（FUSE + 元数据 TiKV + 对象 RGW 的多层往返）**。
   CephFS 同后端随机读立刻 3.8→13.9（3.6×）就是铁证：**瓶颈在 JuiceFS 框架本身**。

### 为什么 JuiceFS 随机读这么差

一次 256k 随机读，JuiceFS 要：
`FUSE 拦截 → 查 TiKV（chunk/slice 位置，一次网络往返） → 向 RGW GET 对象（HTTP，又一次往返）`。
每个随机 IO 都走这条全链路，**IOPS 被往返延迟钉死**，与介质无关。
而随机写较快（38）是因为 JuiceFS **写新 chunk 整块写**，绕过了 EC 的就地改写（RMW）。

---

## 二、根本判断

> **这套架构（JuiceFS → S3/RGW → EC）对随机小 IO 先天不友好**：每个随机 IO 都付出
> "FUSE + 元数据往返 + 对象往返"的固定延迟税。顺序大 IO 能摊薄这个税，随机小 IO 不能。
> 所以随机读优化的核心是**缩短/减少这条往返链路**，而不是再去碰已被排除的硬件/介质。

---

## 三、调优方向（按优先级，含框架内 + 换方案）

### A. JuiceFS 框架内（先做，低成本）

1. **调小 JuiceFS block-size（最可能见效，治随机读放大）**
   - JuiceFS 默认 block 4MB；256k 随机读却要拉取**整个 4MB block** → 读放大 ~16×。
   - 格式化时把 `--block-size` 调小（如 256k~1M），让随机读不必拉整块。
   - ⚠️ 需重新 format 文件系统才能改 block-size。

2. **量化定位随机读瓶颈**（2026-06-12 已诊断，juicefs stats）：

   **测试方法**：fio randread（8 job, bs=256k, direct=1），同时 `juicefs stats <mount>` 采集。

   **juicefs stats 原始输出（2s 间隔）**：

   ```
   ------usage------ ----------fuse--------- ----meta--- -blockcache ---object--
    cpu   mem   buf | ops   lat   read write| ops   lat | read write| get   put 
   29.6%  412M   92M| 683  1.59     0    83M|  13  1.86 |   0     0 |   0    88M   ← 写阶段：FUSE 写入快(1.59ms)
   34.1%  412M   88M| 824  1.49     0   100M|  12  1.80 |   0     0 |   0    64M
   34.1%  413M   80M| 823  0.08     0   100M|  11  1.53 |   0     0 |   0    28M
   26.1%  437M  124M|  64  45.8  2816K    0 |  41  1.19 |   0    16M|  18M   72M   ← 读开始：FUSE 延迟暴涨至 45.8ms
   65.5%  505M  185M|  54   124  4480K    0 |  11  1.00 |3968K   92M| 100M    0    ← 读：FUSE lat 124ms, GET 100M/2s
   78.6%  536M  199M|  41  94.2  3072K    0 |   3  1.08 |  14M  104M| 115M    0
   58.1%  542M  178M|  71   193  6144K    0 |  10  1.01 |  33M   96M| 101M    0    ← 读：FUSE lat 193ms
   70.4%  543M  208M|  84  72.8  8576K    0 |   4  1.21 | 148M   96M| 108M    0
    147%  583M  213M| 440  19.7    52M    0 |   8  1.10 |1043M   96M| 106M    0    ← block cache 命中后 FUSE 延迟降至 19.7ms
   1067%  790M  217M|4104  1.98   510M    0 |  10  0.97 |  10G   92M| 103M    0    ← CPU 吃满，数据来自本地缓存
   ```

   **分析**：
   - **写阶段（前三行）**：FUSE ops 683-824，延迟 0.08-1.59ms，object PUT 28-88M → 写路径快，接近 100 MB/s
   - **读阶段（中间四行）**：FUSE ops 骤降至 41-84（写转为读），延迟暴涨至 45.8-193ms → 每个读请求等待 100ms+
   - **缓存命中后（后两行）**：block cache 命中返回，FUSE ops 飙至 440-4104，延迟降至 19.7-1.98ms
   - **meta ops 始终只有 3-41**，延迟 1ms → TiKV 完全不是瓶颈
   - **object GET 峰值 100-115M/2s ≈ 50-58 MB/s** → RGW 聚合读吞吐（这是通过 JuiceFS FUSE 处理后看到的有效吞吐，不是 RGW 原始吞吐）
   - **CPU 峰值 1067%** → JuiceFS 进程吃满所有核心

   **关键诊断指标汇总**：
   | 指标 | 写阶段 | 读阶段(RGW) | 读阶段(缓存命中) |
   |------|-------|------------|----------------|
   | FUSE ops | 683-824 | 41-84 | 440-4104 |
   | FUSE lat | 0.08-1.59ms | **45.8-193ms** | 1.98-19.7ms |
   | meta ops | 11-13 | 3-11 | 8-10 |
   | meta lat | 1.5-1.8ms | 1.0-1.2ms | 0.97-1.1ms |
   | object GET | 0 | 100-115M/2s | 103-106M/2s |
   | object PUT | 28-88M/2s | 0 | 0 |
   | block cache read | 0 | 4K-148M | **1-10G** |
   | CPU | 29-34% | 58-79% | **147-1067%** |

   **网络**：ping RTT ~0.2ms，不是瓶颈。
    
   **结论**：随机读瓶颈在 **RGW object GET** 与 **JuiceFS FUSE 处理**的合路延迟
   （读阶段 FUSE lat 100-199ms，写阶段仅 1.5ms）。TiKV 元数据不是瓶颈（1ms）。
   缓存命中后吞吐大幅提升（FUSE 延迟从 193ms 降至 1.98ms），但 randrw 场景缓存
   命中率极低。

3. **元数据引擎换 Redis** → ❌ **跳过**：已诊断 TiKV 延迟仅 1ms，不是瓶颈。
   换 Redis 解决不了 RGW object GET 的 100ms+ 延迟。

4. **缓存策略** → ❌ **无效**：已测过去掉 `--direct=1`（12a），无变化（3.7 vs 3.8）。
   randrw 随机偏移、60s 内几乎零复用，缓存命中率极低。
   juicefs stats 也证实——block cache read 在读阶段仅 4K-148M/2s，数据几乎全从 RGW 拉。
   缓存只对重复读取有效，本 workload 不适用。

5. **多客户端 / 多挂载点并发** → ✅ **已测，线性扩展**（2026-06-12）

   **方法**：tikv-node 上 3 个独立 JuiceFS 挂载点，各跑同参数 fio randrw，
   共享同一 S3 后端（ramdisk EC 池）。

   **结果**：
   | 客户端 | 读 | 写 |
   |--------|-----|-----|
   | client1 | 2.7 MiB/s (2.8 MB/s) | 31.8 MiB/s (33.3 MB/s) |
   | client2 | 2.6 MiB/s (2.7 MB/s) | 32.0 MiB/s (33.6 MB/s) |
   | client3 | 2.8 MiB/s (2.9 MB/s) | 32.9 MiB/s (34.4 MB/s) |
   | **聚合** | **~8 MiB/s (~8 MB/s)** | **~97 MiB/s (~101 MB/s)** |
   | 单客户端基线 | 3.8 MB/s | 38 MB/s |

   **分析**：
   - 读写均**线性扩展**（3× 客户端 → ~3× 吞吐），无争抢
   - **写聚合 101 MB/s 已逼近后端天花板**（rados bench 106 MB/s）
   - **读聚合 8 MB/s 仍远低于 59 目标** — 基数太低，按线性需 ~22 客户端不现实
   - 多客户端分摊了 FUSE CPU 压力，但不降低单请求的 RGW 对象延迟（100ms+）

   **结论**：多客户端有效但治标不治本。写入已触顶，读取的根因仍是 RGW HTTP 往返延迟，
   框架内优化空间已挖尽。

### B. 换方案（框架内榨干后认真评估）

| 方案 | 随机读 | 随机写 | 代价/约束 |
|------|-------|-------|----------|
| **CephFS（内核态）替代 JuiceFS** | 13.9（已测） | 13.8 | 无 FUSE 税；但 EC overwrite 触发 RMW 拖慢写；需 POSIX 共享语义 |
| **CephFS + 副本元数据/数据池** | 待测 | 待测 | 副本避开 EC RMW，写可能改善；副本违反 EC 4+2 规格 |
| **JuiceFS 直连 RADOS（去掉 RGW）** | 待测 | 待测 | 少一层 HTTP/RGW 往返，随机延迟可能降；仍有 FUSE+元数据 |
| **Ceph RBD + 本地文件系统** | 待测（通常远高） | 待测 | 块设备随机性能好；但单客户端独占，非多端共享 |

> 选型前提：取决于**业务访问模式**——是否必须多客户端共享 POSIX 文件？
> 若是 → CephFS/JuiceFS；若可块设备独占 → RBD 随机性能最佳。

### C. 已排除（不要在随机场景重复尝试）

- ❌ 换介质 / 更快盘（JuiceFS 随机换内存盘无效，3.8→3.5）
- ❌ 增大 OSD 数 / RAID 调参 / 调度器（顺序阶段已证无效，瓶颈不在后端硬件）

## 四、下一步行动清单

1. **先诊断**：`juicefs stats` + 节点 `ping` RTT + `iperf3`，定位随机读 3.8 卡在元数据还是对象往返。
2. **改 block-size 重测**（2026-06-12 已执行，无效）：
   | block-size | 读 | 写 |
   |-----------|-----|-----|
   | 4M（默认） | 5.7 MB/s | 39.3 MB/s |

   而在每个 I/O 的 FUSE+TiKV+RGW 往返延迟——这个延迟和 chunk 大小无关。
   **A1 方向已排除。** 下一步：A2 诊断 RTT/元数据耗时，定位延迟来源。

## 五、一句话

随机**写 38** 接近目标 59，随机**读 3.8** 差一个数量级是主战场。瓶颈不在硬件/介质
（换内存盘无效），而在 **JuiceFS 的 FUSE + 元数据 + RGW 多层往返**（CephFS 同后端读
即 3.6×）。优先：**①调小 block-size 治读放大 ②诊断 RTT/元数据定位 ③元数据换 Redis /
去 RGW 直连 RADOS ④评估 CephFS/RBD 换栈**。先诊断再动手，别碰已排除的硬件方向。
