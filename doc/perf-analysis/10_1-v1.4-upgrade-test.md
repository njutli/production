# 10_1 JuiceFS v1.4.0-rc1 升级测试（2026-06-20）

> 对比 v1.3.1 与 v1.4.0-rc1 在同池（`juicefs-data` EC 4+2 ec-prod）、同配置（256K block-size、STORAGE=ceph）下的各方面性能。
> v1.4.0-rc1 从 gitee mirror 拉源码、Go 1.26.4 编译（`-tags ceph`）。

---

## 一、编译与部署

```bash
# Go 1.26.4（golang.google.cn）
git clone --depth 1 --branch v1.4.0-rc1 https://gitee.com/juicedata/juicefs.git
CGO_ENABLED=1 go build -tags ceph -o juicefs .
# 部署至 tikv-node / ceph-node1 / ceph-node2
```

v1.4 新增关键参数：`--max-downloads`（默认 200），v1.3.1 无此参数。

---

## 二、测试方法

所有对比在**同一池**（`juicefs-data` EC 4+2 ec-prod）、**同一时间段**串行跑，消除池差异。

| 测试 | 步骤 | 重复 |
|------|------|------|
| 单客户端 128G | `bench-juicefs.sh` SKIP_SEQ=1 全量 | REPEAT=3 |
| 多客户端 32G | `bench-c-multihost-3client.sh` | 单次 |
| `--max-downloads` sweep | 手动 mount + fio 32G layout | 单次 |

## 三、单客户端 128G 对比（同池）

| 测试 | v1.3.1 | v1.4.0-rc1 | 变化 |
|------|--------|-----------|------|
| Layout 写 | 54.3 MB/s | 51.3 MB/s | −5.5% |
| **纯 randread** | **45.7 MB/s** | **44.2 MB/s** | **−3.3%** |
| 纯 randwrite [analysis] | 64.2 MB/s | 64.4 MB/s | +0.3% |
| randrw 读 [analysis] | 28.5 MB/s | 26.2 MB/s | −8.1% |
| randrw 写 [analysis] | 28.1 MB/s | 25.8 MB/s | −8.2% |
| 纯 randwrite [spec] | 65.3 MB/s | 68.5 MB/s | +4.9% |
| randrw 写 [spec] | 29.8 MB/s | 35.2 MB/s | +18.1% |

> 所有差异在 ±10% 以内，单客户端层面 v1.4 与 v1.3.1 **基本持平**（randread −3% 属方差）。
> 之前报告 −41% 的 randwrite 下降系**旧池快/新池慢**的池差异误判。

## 五、多客户端 32G 对比（同池）

| 阶段 | 客户端 | v1.3.1 | v1.4.0-rc1 | 变化 |
|------|--------|--------|-----------|------|
| P1 单 | tikv | 22.5 | **31.4** | **+39.6%** |
| **P2 双** | tikv+node1 | **50.3** | **76.8** | **+52.7% ✅** |
| P3 三 | tikv+node1+node2 | 50.0 | **80.3** | +60.6% |

> - **P2 双客户端 76.8 > 59 目标。** `--max-downloads` 打破单客户端并发瓶颈。
> - v1.3.1 的 P2→P3 不增反略降（50.3→50.0），v1.4 的 P2→P3 仍有增益（76.8→80.3）
> - ceph-node2 在三客户端中性能偏低（20.9 vs 29-30）是 OSD 协存竞争，非版本问题

## 六、结论

### 6.1 单客户端明细（证明退化非波动）

逐 run 数据，在同一 pool 上紧接跑：

| 测试 | 版本 | run1 | run2 | run3 | 均值 | Δ |
|------|------|------|------|------|------|----|
| 纯 randread | v1.3.1 | 44.4 | 46.3 | 46.4 | **45.7** | — |
| | v1.4 | 43.5 | 44.4 | 44.6 | **44.2** | −3.3% |
| randrw 读 [a] | v1.3.1 | 22.6 | 30.3 | 32.6 | **28.5** | — |
| | v1.4 | 21.4 | 28.3 | 28.9 | **26.2** | −8.1% |
| randrw 写 [a] | v1.3.1 | 22.3 | 29.9 | 32.1 | **28.1** | — |
| | v1.4 | 21.1 | 27.9 | 28.4 | **25.8** | −8.2% |

> 每期三个 run，v1.4 逐 run 均低于 v1.3.1——**非偶发波动，v1.4 单客户端确有退化（−3~8%）**。

### 6.2 结论

| 场景 | v1.4 vs v1.3.1 | 解读 |
|------|---------------|------|
| **单客户端** | **−3~8%** | `--max-downloads` 引入调度开销，对单客户端是净损 |
| **多客户端** | **+53~61%** | `--max-downloads` 打破了并发连接瓶颈，数量级增益 |

**判定**：v1.4 是一个 tradeoff——牺牲单客户端小部分性能（−3~8%）换取多客户端数量级提升（+53%）。是否升级取决于是否走多客户端架构：
- 多客户端 → 升级 v1.4，双客户端 76.8 直接达标
- 单客户端 → 保持 v1.3.1，v1.4 反而退步
