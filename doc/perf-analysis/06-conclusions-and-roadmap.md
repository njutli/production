> 16 jobs 多任务顺序写 ≈ 单任务（80~90），延迟却暴涨 15× → 并发不是瓶颈。

### 2. 后端裸能力（rados bench，绕过 RGW/JuiceFS）

| 场景 | 写带宽 | 平均延迟 | 备注 |
|------|-------|---------|------|
| 4 OSD（node2 down，降级） | 108 MB/s | 2.28s | node2 重装前 |

后端压测时（102MB/s）：三节点网卡 64~84%（**都没满**）、盘 util ~50%、CPU 3~8%、
**IOPS 仅 25、平均延迟 2.45s**。

### 3. LB 负载均衡（HAProxy stats）

三 RGW 请求/字节几乎完全均分（rgw1/2/3 ≈ 5980/5979/5979，字节 22.9G/22.98G/22.83G）
→ **LB 分流没问题**。

### 4. 随机负载（256k randrw）

128 并发随机写仅 ~36 MB/s（远低于顺序）。**blktrace 实测对比**（256k 随机读写，
本地 ext4 vs Ceph OSD，与第 5 节顺序场景同法）：

| 落盘指标 | 本地 ext4 (sda) | Ceph-OSD (sdb) |
|---------|----------------|----------------|
| 端到端带宽 | 读152+写154 ≈ 306MB/s | JuiceFS 侧远低（~36 量级） |
| 平均 IO 大小 | 502 扇区 ≈ 257 KB | 215 扇区 ≈ **107 KB**（拆小一半） |
| 顺序性 (seek MODE=0) | 46% | **28%**（更碎） |
| **随机写期间盘上读 (Reads Queued)** | 有读 | **0（几乎无读）** |

- **关键修正**：`05_1` 曾假设"36MB/s 是 EC RMW（部分条带写先读旧数据）导致"，
  但 blktrace 实测 **Ceph OSD 盘上 `Reads Queued ≈ 0`** —— 随机写期间几乎没有读。
  → **EC RMW 读放大假设被实测否定。**
- 随机写慢的机理**与顺序场景相同**：Ceph 把 IO 拆碎（257KB→107KB）、落盘碎片化
  （仅 28% 连续）+ 随机 workload 本身 → HDD 频繁寻道 → 吞吐低。

#### 为什么"本地 ext4 盘上有读，Ceph 盘上几乎无读"？

差别不在"随机 vs 顺序"，而在**操作对象**和**是否经 JuiceFS**：

- **本地 ext4**：fio 先 layout 出文件，`randrw` 在**已存在文件**里随机读+随机写。
  其中随机读真打到盘 + ext4 还要读元数据（inode/bitmap）→ **盘上自然有读**。
- **Ceph 侧盘上无读**，两个原因叠加：
  1. **随机读被 JuiceFS 客户端缓存吸收**：刚 layout 写入的数据还在 JuiceFS 本地
     block cache / 页缓存里，fio 的随机读大多在客户端层命中，**没下沉到 OSD 盘**；
  2. **写不触发 EC RMW**：JuiceFS 把文件切成固定 4MB chunk，**每个 chunk 作为一个
     完整的新 S3 对象整块写入**。即使 fio 层是"随机写文件某个偏移"，JuiceFS 也是
     **整 4MB 对象覆盖写**，而非去 Ceph 里就地改某对象的几 KB。对 Ceph EC 而言，
     写完整对象 = **整条带写（full-stripe write）**，直接算出 6 个分片全部写下去，
     **无需先读旧数据/旧校验** → 不产生 RMW 的读。
- **推论**：这套架构（JuiceFS 对象不可变 + 整块覆盖）天然**避开了 EC RMW 这个坑**，
  代价是**写放大**（改几 KB 也要重写整个 4MB chunk）。所以瓶颈不在"读放大"，
  而在前述"IO 拆碎 + 寻道 + 写放大"。

### 5. blktrace 落盘 IO 模型对比（2026-06-11，铁证）

同样 4M 顺序写，对比"本地 ext4 直写 HDD(sda)" vs "Ceph OSD 落盘(sdb)"，
两块同型号 HDD（PERC H730），用 `tests/blktrace-compare.sh` 抓 40s：

| 落盘指标 | 本地 ext4 (sda) | Ceph-OSD (sdb) | 含义 |
|---------|----------------|----------------|------|
| 下发 IO 数 (#Q) | 18,516 | **38,078** | Ceph 把同样数据拆成 2 倍多的 IO |
| 平均 IO 大小 (BLKavg) | 491 扇区 ≈ **251 KB** | 214 扇区 ≈ **107 KB** | Ceph 落盘 IO 被拆小一半 |
| 合并比 (Ratio) | 1.0 | 2.4 | Ceph 侧 IO 碎、被块层合并 |
| **顺序性 (D2D seek MODE=0 占比)** | **90%** (16680/18516) | **29%** (4605/15854) | **本地几乎全顺序；Ceph 71% 要寻道** |
| 单 IO 延迟 (D2C avg) | 39 ms | 0.11 ms | 见下方解读 |

- **本地 ext4**：90% 的盘 IO 是连续的（seek=0），大块（251KB）→ 真顺序写，磁头几乎不动；
- **Ceph OSD**：只有 29% 连续，**71% 需要寻道**，IO 小（107KB）且数量翻倍 →
  顺序写在盘上被**随机化**。
- 单 IO 延迟反而 Ceph 更低（0.11ms vs 39ms），不矛盾：Ceph 是"小 IO 写进 RAID
  卡缓存就快速 ack"，但**数量太多 + 71% 要寻道**，盘大量时间花在寻道而非传输 →
  整体吞吐被拖低，与 `%util 仅 50%`（盘在寻道/等待而非满速）一致。

> 这组数据**实测推翻了"对象数据基本顺序落盘"的早期推断**：经 Ceph(EC 拆片 +
> BlueStore 数据/元数据/WAL 混写)后，用户的顺序大 IO 确实变成了碎片化、
> 需频繁寻道的小 IO。这是 HDD 上吞吐差几倍的直接原因。

> 注：早期"本地裸盘 fio 408MB/s"是 256k 大块 + RAID 卡缓存 + 写合并(merge 11万次)
> 的**准顺序**成绩，不代表 HDD 真随机能力，不能与 Ceph 后端直接对比。

## 二、当前瓶颈定性（核心结论）

### ✅ 已实测确认

1. **网络不是当前瓶颈**：后端压到 102MB/s 时三节点网卡只有 64~84%，没打满。
   → 换更高网络（10G）**只能让端到端从 87 逼近 102，无法突破 102**。
   （前提：node1 千兆链路保持正常——见下方风险）

2. **客户端并发不是瓶颈**：16 jobs ≈ 1 job 的总吞吐。

3. **单 RGW 入口曾是瓶颈，但已基本解决**：单→双 RGW 提升 9%；
   双→三无提升。说明 RGW 入口已不再是主要限制。

4. **RGW 数量已无收益**：三 RGW 反而略降，LB 也证明分流均匀。
   **不要再加 RGW。**

5. **瓶颈是「Ceph(EC 拆片 + BlueStore)把顺序大 IO 转成碎片化小 IO」+「HDD 寻道」**
   （最关键结论，已由 blktrace 实测坐实）：
   - 决定性证据：后端 102MB/s 时 **IOPS 仅 25、平均延迟 2.45s**，而**盘 util 才 50%、网卡/CPU 都不满**；
   - **blktrace 实测**（同样 4M 顺序写，本地 ext4 vs Ceph OSD，见下方第 1.5 节）：
     本地 ext4 落盘 **90% 连续、IO 251KB**；Ceph OSD 落盘 **仅 29% 连续、IO 107KB、IO 数翻倍**；
   - 即：**用户下发的顺序大 IO，经 Ceph 后变成了碎片化、71% 需寻道的小 IO** —— HDD 最怕寻道，这是吞吐差几倍的实测根因；
   - 裸盘**顺序**写能到 420MB/s，但 Ceph 把它打散成准随机写 → 实际 102MB/s。

### ⚠️ 待验证 / 已排除的推测

- **PG/primary 倾斜到 node1** → **已排除**：拓扑快照显示 primary PG 在 6 个 OSD 上
  分布均匀（15~18% each），容量/PG 数也均匀。
- **node1 网卡走 RGW 时 100%、绕过 RGW(rados bench) 降到 84%** → node1 那份"额外
  流量"是 RGW 路径特有（疑为 bucket index/meta 等 replicated 池写汇聚），
  **但它不是顺序吞吐的主瓶颈**（绕过后端也才 102）。优先级低。
- **128 并发随机 36MB/s 的成因** → **已用 blktrace 验证**：不是 EC RMW 读放大
  （盘上读≈0），而是与顺序同源的"IO 拆碎 + 寻道"（见第 4 节）。

## 三、关于「换方案能否提升」的评估（回应常见疑问）

| 设想 | 能否突破 ~102 顺序上限 | 结论 |
|------|----------------------|------|
| 更高网络（10G/25G） | ❌ 网卡当前没满 | 只能逼近 102，不突破；没必要 |
| 增加 RGW 数量 | ❌ 已实测三 RGW 无收益 | 不做 |
| **磁盘调度器/IO 合并参数调优** | ❌ **已实测无效**（见下方专项） | 不做 |
| **EC 4+2 → 多副本(size=3)** | ❌ **已实测：副本反而慢 16%**（102→86） | 违规**且**退步，彻底排除 |
| 换 MinIO 等轻量网关 | ⚠️ 看 MinIO 后端配置，多数情况无改善 | 见下方专项分析 |
| 增大 EC 池 pg_num（32→128/256） | ⚠️ 可能小幅提升并行度 | **低成本、不违规，值得一试** |
| BlueStore WAL/DB 放 SSD | ✅ 显著改善 HDD 写延迟 | 需少量 SSD，性价比高 |
| **后端 HDD → SSD/NVMe** | ✅✅ 根治（延迟/IOPS 数量级提升） | **唯一能数量级突破的路径**，需硬件投入 |
| 增加 OSD 数量（更多盘并行） | ✅ 提升聚合吞吐 | 需硬件 |

> 关键认知：**瓶颈在「数据落盘」这一层（HDD+EC），不在网关/网络/CPU/RGW 数量。**
> 软件侧已接近榨干，顺序吞吐天花板 ~102MB/s。

### 专项：换 MinIO 等其他后端，能改善吗？

结合 blktrace 结论（瓶颈是"顺序 IO 被拆碎 + HDD 寻道"），分情况：

- **MinIO + 纠删码（默认）**：MinIO 同样把对象切片+校验后写到多块盘，
  落盘 IO 模型与 Ceph EC 类似（甚至 MinIO 的 EC 是每对象独立编码，元数据
  开销不同但仍是"拆片+多盘"）。**在同样的 HDD 上，大概率不会比 Ceph 明显好。**
- **MinIO + 单盘/无 EC（每盘一个 MinIO 实例，JuiceFS 侧做冗余）**：
  落盘更接近"顺序大写"，HDD 上可能更快——**但这就放弃了冗余/规格**。
  注意：这里"更快"指的是**单盘顺序写（无任何冗余）**；若改成 **3 副本**，
  实测反而比 EC 更慢（见 `results-table.md` 第四节，副本写放大 3× > EC 1.5×）。
- **共性**：MinIO 省的是网关/元数据的 CPU 开销，而我们的 **CPU 才用 3~8%**，
  根本不在 CPU 上。换网关解决不了"HDD 寻道"这个真瓶颈。

> 结论：**换 MinIO 不是对症的解法。** 真正决定落盘 IO 模型的是"冗余方式
> （EC vs 副本/单盘）"和"介质（HDD vs SSD）"，而不是"用 Ceph 还是 MinIO"。
> 在 EC + HDD 不变的前提下，换 MinIO 预期无明显改善；若靠 MinIO 放弃 EC 来提速，
> 则和"EC→副本"一样违反规格。**与其换存储栈，不如把精力放在 SSD/WAL-DB
> 这类对症的硬件改动上。**

### 专项：调磁盘调度器/IO 合并参数能提升吗？（2026-06-11 已实测，无效）

设想：既然慢在"IO 碎片化"，能否换调度器/调大队列来提高 IO 合并、减少寻道？
**实测验证：无效。**

实验：三节点 sdb 统一 `mq-deadline` + `nr_requests` 256→1024 + `read_ahead_kb` 128→512，
跑同样 Ceph 顺序写并重抓 blktrace，与调参前对比：

| 落盘指标 | 调参前 | 调参后 |
|---------|-------|-------|
| 平均 IO 大小 | 107 KB | 107 KB |
| 合并比 Ratio | 2.4 | 2.4 |
| 顺序性 (seek=0 占比) | 29% | **29%** |
| D2C 延迟 | 0.11 ms | 0.11 ms |

**所有指标纹丝不动。** 原因：

- 调度器只能合并**磁盘地址相邻**的 IO；而 Ceph 下发的碎 IO **地址本就分散**
  （EC 6 分片散落 + BlueStore 数据/RocksDB 元数据/WAL 写在盘上不同区域），
  不相邻 → 合并不了；
- Ratio 一直 2.4，说明**能合并的块层早已合并**，剩 71% 不连续是 Ceph 布局决定的；
- 瓶颈层（Ceph/BlueStore 数据布局）在块设备调度器**之上**，调度器够不着。

> 结论：**磁盘调度器/合并参数对本场景无效，从优化清单划掉。** 要改变落盘连续性，
> 只能动 BlueStore 层参数（min_alloc_size / deferred write，风险高需重启 OSD）
> 或换 SSD（无寻道，碎 IO 不再有惩罚——根治）。
>
> 附带发现：node3 的 sdb 调度器是 `none`（noop），对 HDD 是次优配置；虽本次证明
> 对 Ceph 吞吐无影响，但若有本地 IO 需求，建议统一为 `mq-deadline`（运维一致性，
> 非性能项）。

### 专项：换其他后端（不止 MinIO）能有效吗？

判断标准：在 **HDD + EC 4+2 两个硬约束不变**的前提下，后端要"有效"必须改变
落盘 IO 模型（不拆片 / 把碎随机写转顺序 / 把元数据小 IO 移出 HDD）。据此评估：

| 后端类型 | 改变落盘 IO 模型？ | 满足 EC4+2 + ≥60% 利用率？ | 预期 |
|---------|------------------|--------------------------|------|
| MinIO / Swift / Ceph（均 EC） | ❌ 仍拆片+多盘 | ✅ | **无改善**（同 Ceph） |
| SeaweedFS volume（追加写，顺序化落盘） | ✅ 顺序追加、不拆片 | ❌ 多为副本=违规 | 快但违规 |
| Ceph/任何后端改 3 副本（size=3） | ⚠️ 不拆片但写放大 3× | ❌ 利用率 33%<60% | **实测反而更慢**（见下） |
| JuiceFS 直挂本地盘 / NFS（绕过对象层、无冗余） | ✅ 绕过 EC、顺序 | ❌ 无冗余 | 快但违规 |
| **任何后端 + HDD + EC4+2** | ❌ | ✅ | **都无改善** |

> ⚠️ 注意区分两种"放弃 EC"：
> - **单盘/追加写（顺序化、无冗余或日志结构）** → 真能更快（改变了 IO 模型为顺序）；
> - **3 副本（size=3）** → **实测反而比 EC 慢 16%**（102 vs 86），因副本写放大 3×
>   > EC 的 1.5×，盘瓶颈下写得多更慢。所以"EC→副本"这条路**违规且退步，彻底排除**。

> **核心结论：只要"EC 4+2 + HDD"两个硬约束同时在，换任何后端都救不了**——
> 落盘仍是碎 IO + 寻道。换后端能提速的前提无一例外是"放弃 EC"（违规）或
> "换 SSD"（介质问题，与后端无关）。
> 因此**不要把精力花在评估/迁移其他对象存储**；真正有效的是：① 换 SSD（保留
> Ceph+EC，根治）；② WAL/DB 上 SSD（保留 Ceph+EC，缓解）。
> 注：**3 副本实测比 EC 还慢**（见 `results-table.md`），只有"顺序追加/单盘无冗余"
> 类才更快但违规；总之换后端/换冗余方式都不是出路，对症的是介质（SSD）。

## 四、后续优化方向（按「成本 / 是否违规 / 收益」排序）

### A. 低成本、不违规、立即可试

1. **增大 EC 池 pg_num**：`default.rgw.buckets.data` 现为 32，对 6 OSD 偏少。
   调到 128 或 256，提升 PG 并行度，复测顺序/随机吞吐。
   ```bash
   ceph osd pool set default.rgw.buckets.data pg_num 128
   ceph osd pool set default.rgw.buckets.data pgp_num 128
   ```
2. **（已完成）EC vs 副本对照**：rados bench 实测 EC 102 vs 副本 86 MB/s
   —— **副本反而慢 16%**（写放大 3× > EC 1.5×，且 JuiceFS 无 RMW）。
   结论：EC→副本违规且退步，已排除，不必再试。
3. **（可选）打散 RGW index/meta 流量**：若想让 node1 走 RGW 时不再 100%，
   查 `default.rgw.buckets.index` 等 replicated 池 primary 分布，用
   balancer / primary-affinity 打散。收益有限（顺序上限仍 102），优先级低。

### B. 中成本、不违规、收益明确

4. **给 BlueStore 配 SSD 作 WAL/DB**：若能加少量 SSD（哪怕每节点一块小 SSD），
    把 OSD 的 WAL/DB 迁上去，能显著降低 HDD 写延迟 → 提升 EC 写吞吐与随机性能。
    性价比最高的硬件改动。
    → **先用内存盘模拟 SSD 验证效果**，步骤见 `06_1-ramdisk-wal-db-test.md`。

### C. 高成本、根治（唯一能数量级突破的路径）

5. **后端介质 HDD → SSD/NVMe**：直接消除寻道延迟，IOPS/延迟数量级改善，
   EC 4+2 在 SSD 上表现远好于 HDD。是突破 102MB/s 顺序、36MB/s 随机的根本手段。
6. **增加 OSD 数量**：更多盘并行，提升聚合吞吐（需机位/硬件）。

### 不做 / 仅作参考

- ❌ 加更多 RGW（已证无收益）
- ❌ 换更高网络（网卡未满，只能逼近 102）
- ❌ 换 MinIO 等网关（瓶颈不在网关）
- ❌ 调磁盘调度器/IO 合并参数（已实测无效，碎 IO 地址分散合并不了）
- ❌ EC→副本（size=3）：**实测比 EC 还慢 16%**（写放大 3×>1.5×），违规且退步，排除

## 五、运维风险提醒（搬迁后新增）

- **node1 网卡曾在搬迁后自动协商掉到 100Mb/s**，导致后端暴跌到 17.6MB/s。
  已用 systemd service `force-eno1-gigabit.service` 强制千兆并持久化，但
  **根因疑在网线/交换机端口物理质量**（RX errors、对端广告 10baseT）。
  → 建议检查/更换 node1 网线与交换机端口，否则可能再次掉速或丢包。
- 监控用 `tests/node-monitor.sh`（三节点采 CPU/网卡 rx-tx/盘）、
  拓扑用 `tests/cluster-topology.sh`（PG/primary/EC 分布）、
  落盘 IO 模型用 `tests/blktrace-compare.sh`（本地 ext4 vs Ceph OSD 的 IO 大小/
  顺序性对比），后端裸能力对照用 `rados bench`（从中立的 tikv-node 发起，绕过 RGW）。
- 做任何优化（加 PG / WAL-DB 上 SSD / 换 SSD）后，**重跑 `blktrace-compare.sh`**
  看 Ceph 侧落盘的"连续性占比"是否从 29% 提升——这是验证优化是否对症的直接指标。

## 六、一句话总结

千兆网络下，顺序吞吐天花板 **~102 MB/s**，端到端已达 87（后端的 85%）。
瓶颈的本质（blktrace 实测坐实）：**Ceph 的 EC 拆片 + BlueStore 把用户的顺序大 IO
转成了碎片化、71% 需寻道的小 IO，而 HDD 最怕寻道** —— 不在网络、RGW 数量、CPU、
网关软件（换 MinIO 也救不了，瓶颈不在网关）。软件侧（多 RGW+LB+调均衡+加 PG）
已接近榨干；要数量级突破，唯有让落盘 IO 不再受寻道惩罚——即 **WAL/DB 上 SSD
（对症、性价比高）或整盘换 SSD/NVMe（根治）**。随机负载（36MB/s）同理。
EC→副本（size=3）实测反而比 EC 慢 16%（写放大 3×>1.5×、JuiceFS 无 RMW），违规且退步，排除。
