# 阶段性结论汇总与后续优化方向（截至 2026-06-11）

> 本文汇总到目前为止**全部实测数据**与**当前瓶颈定性**，并给出后续优化方向。
> 是 `01`~`05_1` 的总收口；新数据（搬迁后掉百兆修复、双/三 RGW、
> 后端 102MB/s 实测）也一并纳入。结论区分「已实测确认」与「待验证推测」。

## 一、完整测试数据汇总

### 1. 端到端（JuiceFS + fio，4M 顺序写，默认挂载无 writeback）

| 配置 | 端到端写带宽 | node1 网卡峰值 | 备注 |
|------|------------|--------------|------|
| 百兆网络时代 | ~8 MB/s | — | node1/node2 网卡 100Mb/s |

| 千兆 + 三 RGW(全部)+LB | 86.7 MiB/s | 100% | **未提升反略降** |

> 16 jobs 多任务顺序写 ≈ 单任务（80~90），延迟却暴涨 15× → 并发不是瓶颈。

### 2. 后端裸能力（rados bench，绕过 RGW/JuiceFS）

| 场景 | 写带宽 | 平均延迟 | 备注 |
|------|-------|---------|------|
| 4 OSD（node2 down，降级） | 108 MB/s | 2.28s | node2 重装前 |

### 3. LB 负载均衡（HAProxy stats）

### 4. 随机负载（历史）

128 并发随机写仅 ~36 MB/s（远低于顺序），疑为 HDD + EC 随机小写放大（待 `05_1` 对照实锤）。

## 二、当前瓶颈定性（核心结论）

### ✅ 已实测确认

1. **网络不是当前瓶颈**：后端压到 102MB/s 时三节点网卡只有 64~84%，没打满。
   → 换更高网络（10G）**只能让端到端从 87 逼近 102，无法突破 102**。
   （前提：node1 千兆链路保持正常——见下方风险）

2. **客户端并发不是瓶颈**：16 jobs ≈ 1 job 的总吞吐。

3. **单 RGW 入口曾是瓶颈，但已基本解决**：单→双 RGW 提升 9%；
   双→三无提升。说明 RGW 入口已不再是主要限制。

4. **RGW 数量已无收益**：三 RGW 反而略降，LB 也证明分流均匀。
   **不要再加 RGW。**

5. **瓶颈是「HDD 机械盘 + EC 4+2 写延迟」的组合**（最关键结论）：
   - 决定性证据：后端 102MB/s 时 **IOPS 仅 25、平均延迟 2.45s**，而**盘 util 才 50%、网卡/CPU 都不满**；
   - HDD 的瓶颈体现在**寻道延迟**而非吞吐占用，所以 `%util` 没到 100% 也已经是瓶颈；
   - EC 4+2 把每个对象拆 6 片写 6 盘、**等最慢分片** ack，放大了 HDD 的延迟劣势；
   - 裸盘**顺序**写能到 420MB/s，但 EC 把写打散成多盘协调写 → 实际 102MB/s。

### ⚠️ 待验证 / 已排除的推测

- **PG/primary 倾斜到 node1** → **已排除**：拓扑快照显示 primary PG 在 6 个 OSD 上
  分布均匀（15~18% each），容量/PG 数也均匀。
- **node1 网卡走 RGW 时 100%、绕过 RGW(rados bench) 降到 84%** → node1 那份"额外
  流量"是 RGW 路径特有（疑为 bucket index/meta 等 replicated 池写汇聚），
  **但它不是顺序吞吐的主瓶颈**（绕过后端也才 102）。优先级低。
- **128 并发随机 36MB/s 是否 EC RMW 硬限制** → 验证步骤见 `05_1`，**尚未真正跑对照实验**。

## 三、关于「换方案能否提升」的评估（回应常见疑问）

| 设想 | 能否突破 ~102 顺序上限 | 结论 |
|------|----------------------|------|
| 更高网络（10G/25G） | ❌ 网卡当前没满 | 只能逼近 102，不突破；没必要 |
| 增加 RGW 数量 | ❌ 已实测三 RGW 无收益 | 不做 |
| **EC 4+2 → 多副本(size=3)** | ✅ HDD 上副本写通常明显快于 EC | **违反存储规格（利用率 33%<60%），仅作参考** |
| 换 MinIO 等轻量网关 | ❌ 数据仍落同样 HDD+EC | 换的是"门口网关"，慢在"HDD 货架"，无效；CPU 本就才 3~8% |
| 增大 EC 池 pg_num（32→128/256） | ⚠️ 可能小幅提升并行度 | **低成本、不违规，值得一试** |
| BlueStore WAL/DB 放 SSD | ✅ 显著改善 HDD 写延迟 | 需少量 SSD，性价比高 |
| **后端 HDD → SSD/NVMe** | ✅✅ 根治（延迟/IOPS 数量级提升） | **唯一能数量级突破的路径**，需硬件投入 |
| 增加 OSD 数量（更多盘并行） | ✅ 提升聚合吞吐 | 需硬件 |

> 关键认知：**瓶颈在「数据落盘」这一层（HDD+EC），不在网关/网络/CPU/RGW 数量。**
> 软件侧已接近榨干，顺序吞吐天花板 ~102MB/s。

## 四、后续优化方向（按「成本 / 是否违规 / 收益」排序）

### A. 低成本、不违规、立即可试

1. **增大 EC 池 pg_num**：`default.rgw.buckets.data` 现为 32，对 6 OSD 偏少。
   调到 128 或 256，提升 PG 并行度，复测顺序/随机吞吐。
   ```bash
   ceph osd pool set default.rgw.buckets.data pg_num 128
   ceph osd pool set default.rgw.buckets.data pgp_num 128
   ```
2. **完成 `05_1` 的 EC vs 副本对照实验**：用临时复制池跑同样随机写，
   实锤 36MB/s 是否 EC RMW 导致（数据支撑"是否值得为随机性能上 SSD"）。
3. **（可选）打散 RGW index/meta 流量**：若想让 node1 走 RGW 时不再 100%，
   查 `default.rgw.buckets.index` 等 replicated 池 primary 分布，用
   balancer / primary-affinity 打散。收益有限（顺序上限仍 102），优先级低。

### B. 中成本、不违规、收益明确

4. **给 BlueStore 配 SSD 作 WAL/DB**：若能加少量 SSD（哪怕每节点一块小 SSD），
   把 OSD 的 WAL/DB 迁上去，能显著降低 HDD 写延迟 → 提升 EC 写吞吐与随机性能。
   性价比最高的硬件改动。

### C. 高成本、根治（唯一能数量级突破的路径）

5. **后端介质 HDD → SSD/NVMe**：直接消除寻道延迟，IOPS/延迟数量级改善，
   EC 4+2 在 SSD 上表现远好于 HDD。是突破 102MB/s 顺序、36MB/s 随机的根本手段。
6. **增加 OSD 数量**：更多盘并行，提升聚合吞吐（需机位/硬件）。

### 不做 / 仅作参考

- ❌ 加更多 RGW（已证无收益）
- ❌ 换更高网络（网卡未满，只能逼近 102）
- ❌ 换 MinIO 等网关（瓶颈不在网关）
- 📌 EC→副本仅作"放宽规格能快多少"的参考，**违反 EC 4+2 + ≥60% 利用率规格**

## 六、一句话总结

千兆网络下，顺序吞吐天花板 **~102 MB/s**，端到端已达 87（后端的 85%）。
瓶颈是 **HDD 机械盘 + EC 4+2 的写延迟**（IOPS 25 / 延迟 2.45s 是铁证），
**不在网络、RGW 数量、CPU、网关软件**。软件侧（多 RGW+LB+调均衡+加 PG）
已接近榨干；要数量级突破，唯有**后端上 SSD/NVMe 或加 OSD**——但这些超出
当前软件调优范畴，属硬件投入决策。随机负载（36MB/s）同理，需 SSD 根治。
