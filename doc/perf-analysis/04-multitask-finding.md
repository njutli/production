# 多任务并发测试发现（2026-06-09）：瓶颈在单 RGW/单链路，不是并发度

> 数据来源：`results/20260609-155708-baseline.txt`，单 RGW（node1:8000）、
> 千兆网络、JuiceFS 默认挂载（无 writeback）。
> 结论：**聚合吞吐已触到一个全局天花板，且天花板不在客户端并发上——
> 强烈指向单 RGW / node1 单条千兆链路是吞吐瓶颈。**

---

## 一、关键数据

官方顺序大文件测试（4M 块，默认挂载，无 writeback）：

| 测试 | 并发(jobs) | READ | WRITE |
|------|-----------|------|-------|
| 单任务顺序 | 1 | 83.7 MiB/s | 80.1 MiB/s |
| **多任务顺序** | **16** | **91.2 MiB/s** | **75.7 MiB/s** |

并发从 1 → 16，**整体带宽几乎不变**（读 83.7→91.2 微涨，写 80.1→75.7 略降）。

延迟对比（clat）：

| | 单任务 | 多任务(16) |
|---|--------|-----------|
| 平均 clat | ~45 ms | **~690 ms** |
| 最大 clat | ~330 ms | **10000+ ms** |

> 16 个任务总量没涨，但每个任务延迟暴涨 ~15 倍 —— 典型的
> "入口被单一共享瓶颈卡住、请求在排队"的特征。

---

## 二、对调优方向的意义

### 1. 排除了"客户端并发不足"

之前的悬念：单线程顺序只有 80 MB/s，是不是单线程打不满后端？
现在加到 16 并发**总量仍是 80~90 MB/s** → **提高客户端并发救不了**，
瓶颈不在客户端侧。

### 2. 瓶颈是被所有任务共享的单一资源

16 个任务挤在一起还是 80~90 MB/s，说明它们在抢同一个点。结合拓扑，
最可能是：

- **单个 RGW（node1:8000）**：所有任务的 S3 流量都走这一个 radosgw 进程；
- 或 **node1 那条千兆上行链路**本身（80~90 MB/s ≈ 700 Mb/s，接近单条
  千兆的有效吞吐）。

### 3. 顺序写卡在 ~80MB/s，基本排除盘瓶颈

- HDD 顺序写很快（裸盘实测 420 MiB/s），EC 顺序写也远不止 80 MB/s。
- 顺序大文件场景**不该被 HDD/EC 限制**，却卡在 ~80MB/s ≈ 单条千兆量级
  → 瓶颈在 **node1 这一个入口**（RGW 进程或链路），不是 OSD 盘。
- 对比 128 并发随机测试（仅 36MB/s）：那个低值更可能是 HDD 随机写放大；
  但**吞吐型顺序负载的瓶颈是入口**，两类负载瓶颈不同，要分开看。

### 4. 这是"加 RGW + LB 会有效"的第一个正面证据

把 02/03 文档里"加 RGW 是否有效"从**待验证**推进到**很可能有效**
（至少对吞吐型负载）：顺序写不该被 HDD 限制却卡在单千兆量级，说明
瓶颈是单入口；多 RGW + LB 把流量分到多节点多链路，应能突破。

---

## 三、实锤验证步骤（部署多 RGW 前必做）

光看 fio 还不能 100% 断定是"RGW 进程"还是"链路"。**重跑这个 16 并发
测试，同时在三节点跑 `tests/node-monitor.sh` 采样**，看 node1：

```bash
# 三个存储节点上各自启动（压测前）
bash tests/node-monitor.sh 2 sdb            # 或 --source sysstat 用 iostat

# 客户端重跑 16 并发顺序写/读（results/20260609-155708-baseline.txt 同款）
# 结束后三节点 Ctrl-C，看小结
```

判读：

| node1 观察到 | 结论 | 是否该加 RGW |
|-------------|------|-------------|
| `net%` 打满 ~100%（单方向千兆） | 瓶颈是 node1 入口**链路** | ✅ 加 RGW 分到多节点链路必然有效 |
| `rgw%` 吃满若干核、网卡没满 | 瓶颈是单 **radosgw 进程** | ✅ 加 RGW 分摊请求有效 |
| 三节点 `disk%` 都很低 | 进一步排除盘瓶颈 | （顺序写预期如此） |

只要 node1 的网卡或 radosgw CPU 其一打满，即实锤瓶颈在单入口，
可放心部署 node3 的 RGW + LB（改动已就绪，见 `deploy-lb.sh`、
`03-env-change-2026-06.md` 第四节）。

---

## 三点五、验证结果（2026-06-09 实测，已实锤）

### 结果 1：16 jobs 顺序写 + 三节点 node-monitor —— 实锤单 RGW 入口瓶颈

| 节点 | 网卡占用 | OSD盘 util | CPU | 角色 |
|------|---------|-----------|-----|------|
| **node1** | **100%** | 40% | 20% | 唯一 RGW（入口+扇出） |
| node2 | 55% | 41% | 4% | 纯 OSD |
| node3 | 61% | 44% | 5% | 纯 OSD |

- **node1 网卡打满 100%，node2/node3 只有 55~61%** —— 因为 RGW 只在 node1，
  所有客户端写流量先汇聚到 node1，再由它做 EC 编码、把 4 个分片扇出到
  node2/node3。node1 网卡承担"收整份 + 发两份分片"，天然最先到顶；
  node2/node3 只收各自那份分片，所以低。
- **三节点盘 util 仅 ~40%、CPU 都很闲** → 排除 HDD/EC 盘瓶颈、排除 RGW
  进程算力瓶颈。
- **结论：端到端瓶颈就是 node1 这一个 RGW 入口的千兆链路。**

### 结果 2：rados bench 对照（绕过 RGW）—— 量出后端裸能力

`sudo rados bench -p default.rgw.buckets.data 30 write -t 64`（node2 发起）：

```
Bandwidth (MB/sec):  108.088     Max: 124   Min: 0   Stddev: 25.36
Average Latency(s):  2.279
```

| | 写带宽 |
|---|-------|
| 后端裸能力（rados bench，绕过 RGW） | **108 MB/s** |
| 端到端（fio 走单 RGW node1） | **~80 MB/s** |

- **后端 108 > 端到端 80 → 单 RGW 这一层吃掉了约 28 MB/s**，再次印证
  单 RGW 是端到端限制项，绕过它后端更高。
- 但 **rados bench 从 node2 单点发起，它那条千兆同样是约束**，且后端受
  HDD + EC 4+2 分片分发制约（Max 124、Min 0、Stddev 25 抖动明显）——
  所以 **108 这个数也带"单链路 + EC 后端"的天花板色彩，约等于当前硬件
  下后端的吞吐量级**，不是可无限拉高的上限。

### 由此得到的提升预期（重要）

- **加 node3 RGW + LB 有效**：流量分到两个入口、两条千兆，端到端有望
  从 ~80 提升到接近后端的 ~108。
- **但有封顶**：后端本身也就 ~108 MB/s 量级，所以加 RGW 后端到端
  **大概率顶在 ~100~120 MB/s，不会翻倍以上**。指望靠多 RGW 冲到 200+
  不现实——那要后端（更多 OSD / SSD / 更高网络）一起升级。

---

## 四、小结

1. 多任务（16）整体带宽 ≈ 单任务（80~90 MB/s）→ **并发不是瓶颈**。
2. 延迟随并发暴涨 15 倍 → **单一共享入口在排队**。
3. 顺序写卡在单千兆量级 → 瓶颈是 **单 RGW / node1 链路**，**不是盘**。
4. **node-monitor 实锤**：node1 网卡 100%、其余节点 55~61%、三节点盘/CPU
   都闲 → **端到端瓶颈 = node1 单 RGW 入口链路**。
5. **rados bench 对照**：后端裸能力 108 MB/s > 端到端 80 MB/s → 单 RGW
   吃掉约 28 MB/s，**加 RGW+LB 有效**；但后端上限也就 ~108 量级，
   **预期端到端提到 ~100~120 MB/s 封顶**，再往上需升级后端硬件。
6. → 下一步：部署 node3 RGW + LB，复测验证端到端是否提到 ~100+、
   node1 网卡是否降下来、node3 网卡是否升上去。
