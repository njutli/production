## 一、当前已固化的结论（08 系列调试成果）

### 1. 三大已验证结论
| 结论 | 证据 | 量级 |
|------|------|------|
| **保持卷 block-size 与 fio bs 一致（4M→256K）→ 随机读大幅提升** | `08_2` 五之三/四、`08_1` 四之二（双人重测互证） | 纯随机读 **9.8/13.2 → 45.9 MB/s（4.7× / 3.5×）**，达目标 59 的 **78%** |
| **不使用 RGW（直连 RADOS）→ 部分提升** | `08_1` 四 / 四之二 | 纯随机读 +34%、顺序写 +55%（非数量级，RGW 非随机读根因） |
| **顺序读写已对齐后端裸能力，随机写已达标** | 本文二 | 顺序写 117 ≥ rados bench EC 裸能力 ~102（`06`）；随机写 ~56–66 ≈ 目标 59 |

### 2. 已排除的非瓶颈（不再尝试）
| 嫌疑 | 判定依据 |
|------|----------|
| 网络 | `03` 千兆；`06` 后端压满时网卡仅 64–84% |
| 裸盘/介质 | `06` 盘是 SSD，4K 随机读 92,500 IOPS |
| RGW HTTP 层 | 去 RGW 随机读只 +34%，非数量级 |
| JuiceFS 客户端并发/缓冲/预读参数 | `08_2` A/B：object get_c 50–124 不低，调 buffer-size/prefetch/readahead **无一提升**（buffer 反降） |
| 元数据（TiKV） | 延迟仅 ~1ms |
| 读缓存凑数 | 验收口径 cache=0 冷态真值，缓存命中是"热态幻觉"，不采用 |

### 3. 已定位的随机读瓶颈层次（256K 卷上）
1. **主因（已大幅缓解）**：FUSE block 对随机小 IO 的**读放大**——4M 卷 ~16×，256K 卷降到 ~2.5×（残余）。
2. **次因 A：单客户端单张千兆网卡入向带宽**——`08_2` 四之二实测：256K 单客户端 randread 时
   **网卡 RX≈113 MB/s ≈ 91% 线速**，即便后端无限快，单台千兆客户端也拉不动更多。
3. **次因 B：后端 EC 池随机小 IO 上限**——目前看仍有余量（跨机 C 时 OSD TX 未打满），但**尚未直接标定**。

## 二、为什么说"顺序到顶、只剩随机读"——以及它的盲点

- **顺序**：fio 顺序写 256K 卷 **117 MB/s**，已追平/略超 `06` 记录的 rados bench EC 池裸能力 **~102 MB/s**。
  顺序通道在软件层已无空间，天花板是 EC 协议 + BlueStore 软件栈本身（裸盘 178 → EC 102，效率 57%）。
  → **顺序读写不必再调**（除非动后端架构，收益存疑、优先级低）。
- **随机写**：三组 ~56–66 MB/s，已接近/达标（目标 59）；randwrite[优化]口径 RADOS/256K 达 124+，
  后端随机写能力远够。→ **随机写不必再调**。
- **盲点（关键）**：顺序能说"到顶"是因为有 rados bench 顺序数对照；**但随机读从未用 rados bench
  在 `juicefs-data`(EC 4+2) 池上标过裸上限**（`08`/`01`/`diag.sh` 只有命令、无结果，且打的是别的池）。
  没有这个数，就无法判断 45.9 离后端物理极限还有多远。

> **所以"只关注随机读"成立，但第一步不是继续调 JuiceFS（参数已证无效），
> 而是先把随机读的后端裸上限标出来——它决定瓶颈到底还在软件层还是后端已到顶。**

## 四、下一阶段调优方向（按优先级）

### ★ 步骤 1（必做，第一优先）：标定随机读的后端裸上限
- **动作**：在 `juicefs-data`(EC 4+2) 池上跑 `rados bench -p juicefs-data 60 rand`（绕过 JuiceFS/FUSE/RGW），
  并配套 4K/256K 随机读 IOPS/带宽；同机量网卡。
- **回答**：256K 卷的 45.9 离后端随机读物理极限有多远。
  - 若 rados bench rand ≈ 45–55 → **后端 EC 随机小 IO 就是天花板**，软件层已榨干，提升只能动架构（步骤 3）。
  - 若 rados bench rand ≫ 45.9（如 80+）→ **JuiceFS 读路径/客户端网卡仍是瓶颈**，走步骤 2。
- **成本**：极低（~5–10min），环境干净即可跑，不影响任何卷。

### ★ 步骤 2（按验收口径二选一）：把单客户端 45.9 推向 59，或走多客户端聚合
- **2A 降残余读放大（单客户端路线）**：256K 卷 RX≈2.5× 有效读，残余放大来自 EC 取片 + FUSE 路径。
  - 测 **EC 4+2 vs 副本(size=3)** 的 256K randread 对照，**定位**残余放大是否来自 EC 取片
    （仅定位用；生产受"EC 4+2 且利用率≥60%"硬约束，副本不能上生产）。
  - 可选测 **128K block**（冷态）看能否再降放大——注意请求 256k 对 128K 仍 2× 放大、对象数 ×32，未必划算。
  - 核心原则：**只认 cache=0 冷态真值**。
- **2B 多客户端聚合（若验收允许聚合口径）**：跨机 C 已证 1.54×，**2–3 台网卡干净的客户端即可越过 59**。
  - ⚠️ 前提：客户端机**网卡不承载 OSD/RGW**；当前除 tikv-node 外都是 OSD 主机，需独立客户端机。
- **⛳ 待用户确认（决定 2A vs 2B）**：**验收带宽是「单客户端」还是「多客户端聚合」口径？**
  是否有/可加网卡不承载 OSD 的独立客户端机？

### 步骤 3（上生产前必做）：256K block-size 副作用精测 → 定生产 block-size
- 256K 卷写满 100–200GB 真实数据，采集：`ceph df` 池 OBJECTS 数（理论 ×16）+ STORED/USED；
  JuiceFS 元数据规模（`juicefs status`/TiKV key 量级/客户端内存）；大目录 `ls`/`info` 元数据延迟。
  同量 4M 卷对照。⚠️ 在卷**存活时**采集（bench destroy 会清零）。
- 判定：膨胀可承受 → 256K 上生产；元数据延迟恶化/资源吃紧 → 退 **512K**（随机读 ~20、对象数 ×8 折中）。
- 这是**上生产决策的唯一缺口**。

### 步骤 4（备选 / 低优先级）
- **CephFS 备选**（方向二）：随机密集负载用内核态 CephFS（无 FUSE 放大，已测 13.9），
  新建 `tests/bench-cephfs.sh`；作"放宽规格换性能"备选，仅在 JuiceFS 256K 仍不达标时启用。
- **BlueStore 调参**（方向四）：EC 软件栈引擎参数，预期收益存疑，仅作未勾选待办闭环，最低优先级。

## 五、本阶段建议起点

**先做步骤 1（rados bench 标随机读后端裸上限）**——成本极低、决定后续是"调软件"还是"动架构"，
是整个下一阶段的分叉点。环境当前干净，可立即执行。

之后据步骤 1 结果与「验收口径」确认，再决定走 2A / 2B，最后以步骤 3（副作用精测定生产值）收尾。
