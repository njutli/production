## 〇、测量纪律（性能定位通用铁律，踩坑沉淀）

> 这几条由 `08_2` 四之二"OSD 主机收发互抢网卡"的误判复盘而来——那条结论物理上听着合理、
> 却跟自己表里的实测数自相矛盾,直到复核才发现。沉淀为纪律,后续任何定位都先过这几关。

1. **网卡必须分方向看(RX / TX 各自比线速),严禁求和。**
   千兆全双工下 RX 和 TX 是两条独立 ~124 MB/s 通道,互不占用。
   ❌ 错误判据:`RX 86 + TX 56 = 142 > 124 → 互抢`(这正是被推翻的那条)。
   ✅ 正确判据:**任一单方向 < ~110 MB/s(线速 88%)** 才算未触线;某方向触线才是该方向的瓶颈。
   推论:一台机器身兼数职(OSD+客户端)未必互相拖累,关键看各角色压的是哪个方向、是否撞在同一方向。

2. **任何因果结论落地前,先用现有实测数自检能否解释。**
   "互抢"之所以是硬伤,不在物理,而在它跟同一张表里 `OSD主机客户端 28.1 > 干净客户端 26.7` 直接矛盾——
   若真互抢,OSD 主机当客户端应**明显更低**。
   → 每条结论写下前,回头核对它能否解释手上那几个数;**自相矛盾的解释一律推翻重来**。

3. **只认 cache=0 冷态真值。** 读缓存命中是"热态幻觉",不作验收、不作定位依据(见二)。

4. **横向对比必须同口径同工作集。** randrw `--create_on_open` 边写边读是口径病态(short-read 失真),
   横向比一律用 [analysis] 复用布局口径;正式结论用 128G 全口径,轻量 32G 仅作快速探路。

## 二、为什么说"顺序到顶、只剩随机读"——以及它的盲点

- **顺序**：fio 顺序写 256K 卷 **117 MB/s**，已追平/略超 `06` 记录的 rados bench EC 池裸能力 **~102 MB/s**。
  顺序通道在软件层已无空间，天花板是 EC 协议 + BlueStore 软件栈本身（裸盘 178 → EC 102，效率 57%）。
  → **顺序读写不必再调**（除非动后端架构，收益存疑、优先级低）。
- **随机写**：三组 ~56–66 MB/s，已接近/达标（目标 59）；randwrite[优化]口径 RADOS/256K 达 124+，
  后端随机写能力远够。→ **随机写不必再调**。
- **盲点（已解决）**：顺序能说"到顶"是因为有 rados bench 顺序数对照；随机读已用 rados bench
  在 `juicefs-data`(EC 4+2) 池上标了裸上限：
  单节点 **120 MB/s**，2 并发 **154 MB/s**（rados bench 4M 对象，`09_1` 步骤 1）。
  → **后端有余量，瓶颈在客户端侧**（`09_1` 步骤 1、`10_2` 第五节）。

> **所以"只关注随机读"成立，但第一步不是继续调 JuiceFS（参数已证无效），
> 而是先把随机读的后端裸上限标出来——它决定瓶颈到底还在软件层还是后端已到顶。**

## 七、口径澄清：为什么不能"按 3 个 ceph 节点 × 59 = 177"

> 来自一次讨论:既然集群有 3 个 ceph 节点,半速指标是否该 ×3 到 177、当前裸能力 118 是否只到集群 1/3?
> 结论:**不能 ×3,177 没有对应的物理实体,后端 118 也不是 1/3 而是 3/3。** 拆解如下。

### 1. 118 已经是"3 节点 6 OSD 合力"的总数,不能再乘 3

`rados bench -p juicefs-data rand`（`09_1` 步骤 1）打的是**整个 EC 4+2 池**。
数据按 EC 摊在全部 3 节点 / 6 OSD 上,所以测出的 **118 MB/s 本身就是三节点一起干出来的聚合总能力**,
不是单节点成绩。
- ❌ "裸能力只到集群 1/3、还差到 177" —— **不成立**,118 不是 1/3,就是 3/3。
- ❌ ×3=177 在后端**没有对应物理实体**:集群没有第二、第三个独立的 118 等着相加。
- 类比:118 是"一个水池 3 个进水口一起放水的总流量",那 3 个口已经都算进 118 了,不能再 ×3。

### 2. 唯一能 ×3 的对象是"客户端网卡管子",但它不是瓶颈

能 ×3 的只有**客户端网卡入向**:3 台客户端 × ~113 MB/s RX ≈ 340 的"入口管子"。
但管子粗 ≠ 能灌满,实际取两个天花板的较低者:

```
聚合有效读 ≈ min( 客户端入向总带宽 Σ(各客户端 RX) ,  后端裸读 ÷ 读放大 )
              = min( ~340 ,  118 ÷ 2.5 ≈ 47 )
              ≈ 47
```

后端 118 经 256K 卷残余 ~2.5× 放大后,**有效读上限只有 ~47**,远低于客户端管子 340。
→ 瓶颈死死卡在后端 118,加客户端只能逼近 47~55,**到不了 177**。
实测自洽:跨机 2 客户端聚合 **54.8（1.54× 而非 2×）**,边际已在掉,正是逼近 `118÷放大` 上限的表现。

### 3. 真实差距不在"节点数",在"读放大"

| 设想 | 真实 |
|------|------|
| 后端只到集群 1/3,差到 177 | 后端 118 是满的 3/3;177 是把"客户端管子"误当瓶颈 |
| 堆节点 ×3 拿增量 | 增量在 **后端 118 ÷ 放大**:放大 2.5×→有效 47;若压到 1.2×→有效 ~98 |

- **真正该追的差距 = 把残余读放大从 2.5× 往 1.2× 压**（在 FUSE 读路径里,不在堆节点）。
- 后端 118 本身能否再抬（EC 协议 / BlueStore / 网络往返）属**动架构**,收益存疑、低优先级。

### 4. 对验收口径的含义

- 若验收按**单客户端**：目标固定 59,与节点数无关；单客户端被自身千兆 RX 卡死（91% 线速）,
  乘 3 无物理意义（它收不进 177）。
- 若验收按**多客户端聚合**：天花板 ≈ `min(Σ客户端RX, 后端118÷放大)` ≈ 47~55 量级（取决于残余放大）,
  **仍不是 节点数×59**。要冲更高,只能先降放大或抬后端 118。
