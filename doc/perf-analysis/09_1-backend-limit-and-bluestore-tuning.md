## 二、步骤 1（必做）：随机读后端裸上限标定

### 2.1 基准

```bash
# rados bench -p juicefs-data 60 rand
```

| 指标 | 值 |
|------|----|
| 写（`write --no-cleanup`） | **107.3 MB/s**（25 IOPS，延迟 0.64s） |
| **随机读（`rand`）** | **117.7 MB/s**（29 IOPS，延迟 0.54s） |

### 2.2 与 JuiceFS 上层的对照

| 层级 | 随机读带宽 | 倍数 |
|------|-----------|------|
| rados bench（后端裸能力） | **117.7 MB/s** | 1× |
| JuiceFS 256K 卷 randread（冷态 128G） | **45.8 MB/s** | **后端 2.6×** |
| JuiceFS 4M 卷 randread（冷态 128G） | **13.3 MB/s** | 后端 8.8× |

### 2.3 结论

**后端随机读裸能力 118 MB/s ≫ JuiceFS 256K 卷 45.8 MB/s（2.6×）** → 按 `09` 步骤 1 的判别逻辑：

> 若 `rados bench rand ≫ 45.9`（如 80+）→ **JuiceFS 读路径/客户端网卡仍是瓶颈，走步骤 2**。

后端仍有大量余量，瓶颈不在 Ceph EC 池随机小 IO 处理能力上。这与 `08_2` 四之二跨机 C 实验的网卡监控结论互证：**单客户端千兆网卡入向已达 ~113 MB/s（≈91% 线速）**——256K 卷下读放大残余 + EC 取片使 RX≈2.9× 有效读（113 / 45.8 ≈ 2.5× ≈ 08_2 估算的 ~2.5×）。

> 同时对比 `06` 阶段 rados bench 顺序写 102 MB/s（不同时间/负载波动）——本次 107 在误差带内一致。顺序写本就已对齐裸能力，无新信息。

## 三、步骤 4（低优先级）：BlueStore 调参

### 3.1 测试参数与结果

每次 `ceph config set osd` 后立即 `rados bench`（write 60s + rand 60s），无 overlap、无污染。读写同池 `juicefs-data`。

| 参数 | 值 | Write (MB/s) | Rand (MB/s) | vs 基线 |
|------|-----|-------------|-------------|---------|
| **基线** | 全部默认 | **107.3** | **117.7** | — |
| `bluestore_max_blob_size_ssd` | 512K | 104.5 | 117.8 | 写 −2.6%，读 = |
| `bluestore_max_blob_size_ssd` | 1M | 101.7 | 118.1 | 写 −5.2%，读 = |
| `bluestore_prefer_deferred_size_ssd` | 64K | 100.6 | 118.1 | 写 −6.2%，读 = |

> 每档测试后即时回滚，确保互不串扰。

### 3.2 结论

**全部无效。** `rados bench` 写 102–107 MB/s、随机读恒定 ~118 MB/s，BlueStore 引擎参数无论放大 blob size 还是开启 deferred write，均无正向收益（写甚至略降）。

原因：EC 4+2 池的瓶颈是**多节点网络路径**（EC 分片需从多 OSD 收发），不是单个 OSD 的本地 BlueStore 分配/写入策略。`rados bench` 从 ceph-node1 单点发起，千兆网卡已触顶（写 107 ≈ 86% 线速、读 118 ≈ 95%），BlueStore 参数改变不了网络端的硬约束。

> 这与 `09` 步骤 4 的预判完全一致（"预期收益存疑，最低优先级"）。与 `06` 阶段记录的多项后端调优（磁盘调度器 / PG / WAL-DB / 副本 / 内存盘集群）共同构成一致证据链：**Ceph EC 软件栈下的单客户端吞吐天花板由网络协议开销决定，底层引擎参数无法突破**。

## 四、步骤 2A：EC 4+2 vs 副本(size=3) — 残余读放大定位

### 4.1 目的

256K block-size 已将 16× 块级放大消除，但 NIC RX 仍有 ~2.5× 残余放大。两个嫌疑：
- **EC 4+2 多 OSD 取片**（每个 256K 对象需从 4 个 OSD 拉分片，连接/协议开销）
- **JuiceFS FUSE 路径本身**（请求序列化、元数据往返、块边界 misalignment）

> 方法：在 **副本(size=3) 池**上跑同口径 256K randread，与 EC 对照。若副本 NIC RX 显著降低 → EC 取片是主因；若不变 → FUSE 路径是主因。

### 4.2 测试配置

| 项 | EC | 副本 |
|----|----|------|
| Ceph 池 | `juicefs-data`（EC 4+2） | `juicefs-rep`（副本 size=3，新建） |
| 卷 block-size | 256K | 256K |
| 工作集 | 128 jobs × 1G = 128G | 同 |
| fio | bs=256k, iodepth=128, numjobs=128, direct=1, runtime=60s | 同 |
| 缓存 | cold（cache-size 0, drop_caches） | 同 |
| REPEAT | 3 | 3 |
| NIC 监控 | tikv-node `sar -n DEV 1` | 同 |

### 4.3 结果

| 指标 | EC 4+2 (256K) | 副本 size=3 (256K) | 差异 |
|------|--------------|-------------------|------|
| Layout 写带宽 | 108 MB/s | 46.6 MB/s | −57%（副本写放大 3× vs EC 1.5×） |
| **纯 randread（3-run 均值）** | **45.7 MB/s** | **46.0 MB/s** | **≈持平（+0.7%）** |
| randread run 明细 | 45.5 / 45.8 / 45.9 | 45.9 / 46.0 / 46.0 | — |
| NIC RX >100MB/s 均值 | 117.2 MB/s | 118.6 MB/s | ≈相同 |
| NIC RX >100MB/s 采样点 | 181 秒 | 182 秒 | ≈相同 |
| NIC RX/有效读 比 | ~2.5× | ~2.5× | **完全一致** |

### 4.4 结论

**EC 取片不是残余读放大的来源。** 换副本后 NIC RX 和 randread 带宽完全不变——2.5× 残余放大来自 JuiceFS FUSE 路径本身（请求序列化、元数据往返、FUSE 内核态/用户态切换开销），与后端池类型无关。

> 加上步骤 1 已确认后端有余量（rados bench rand=118），**三个瓶颈因子全部锁定到 JuiceFS FUSE 客户端侧**：
> 1. 千兆网卡入向已到 91% 线速（硬顶）
> 2. FUSE 路径 ~2.5× 残余放大（软件顶）
> 3. 后端 118 MB/s 尚有余量（非瓶颈）

**2A 路线判定**：单客户端路线空间已耗尽。256K block 消除了块级放大，EC vs 副本对照排除了多 OSD 取片放大，残余 2.5× 在 FUSE 层不可通过换池/调参消除（`08_2` 已证 buffer-size/prefetch/readahead 全部无效）。

## 六之二、步骤 3（正式重测）：pool 清零 + 冷缓存（2026-06-19）

### 6.2.1 改进

- 删除并重建 `juicefs-data` 池（清空所有历史对象）
- 每档 layout 后 **unmount + wait 65s + remount** 清空 JuiceFS 内部元数据缓存
- 三档在完全一致的集群状态下串行跑

### 6.2.2 正式结果

| 指标 | 4M | 256K | 512K | 256K/4M | 512K/4M | 理论值 |
|------|-----|------|------|---------|---------|--------|
| **Ceph OBJECTS** | 53.39k | 537.97k | 286.47k | **10.1×** | **5.4×** | 16× / 8× |
| **STORED** | 128 GiB | 128 GiB | 128 GiB | = | = | — |
| **USED（EC 后实占）** | **192 GiB** | **192 GiB** | **192 GiB** | **完全相同** | **完全相同** | — |
| **EC 比率** | 1.50× | 1.50× | 1.50× | = | = | — |
| **FUSE RSS（layout 完）** | 2550 MB | **1004 MB** | 1072 MB | **0.39×** | 0.42× | — |
| **cold ls -lR（128 文件）** | 0.017s | 0.017s | 0.019s | = | = | — |
| **cold juicefs info** | 0.392s | 0.417s | 0.394s | = | = | — |
| **juicefs destroy 对象数** | 53,396 | 538,017 | 286,528 | 10.1× | 5.4× | — |
| **Layout 写带宽** | 60.0 MB/s | 66.9 MB/s | 59.0 MB/s | +11% | −2% | — |

### 6.2.3 关键发现

1. **USED 存储量三档完全相等（192 GiB）**：EC 4+2 开销 (1.50×) 不随对象大小变化。**256K 零存储代价。**
2. **OBJECTS 膨胀低于理论值**：10.1×（理论 16×）/ 5.4×（理论 8×）。元数据/索引对象（固定数）稀释了比例。
3. **256K RSS 反而最低**：1004 MB vs 4M 的 2550 MB。4M 的更大 block buffer 占用更多客户端内存——这是 256K 的意外优势。
4. **元数据操作（ls / info）三档无差异**：128 文件规模太小，数百万 chunk 的开销在文件级操作上不可见。
5. **destroy 对象数与 ceph df OBJECTS 完全一致**：数据可信。

### 6.2.4 结论

**256K block-size 无副作用。** 存储空间零膨胀、内存反而更低、元数据操作无差异、OBJECTS 膨胀低于理论值。上生产定 **256K**，无需考虑 512K 折中。

## 七、步骤 2B：多客户端聚合 + 后端裸能力标定（2026-06-19）

> 完成 `09` 步骤 2B：双节点 `rados bench rand` 测后端聚合、三客户端 JuiceFS randread 测端到端聚合。

### 7.1 后端裸能力：双节点 rados bench

| 发起节点 | rand MB/s | 聚合 |
|----------|----------|------|

### 7.3 历史全部多客户端数据对照（32G 256K 口径）

### 7.4 瓶颈链（精确定量）

```
┌─ 客户端 NIC ─┐   ┌─── 后端 ───┐   ┌─ 读放大 ─┐
│ P3 tikv RX=58 │ → │ 145 MB/s   │ ÷ │ 2.5×      │ = 有效读上限 58 MB/s
│ P3 node1 RX=78 │   │(双节点     │   │(FUSE+TiKV  │
│ P3 node2 RX=86 │   │ rados聚合) │   │ +EC+TCP)   │
│ 总 RX = 221   │   └────────────┘   └───────────┘
└───────────────┘
P3 实测聚合 = 55.6 ≈ 58 → 距天花板仅差 4%
```

### 7.5 残余读放大分解（2.5× 的来源）

256K block-size 已消灭块级放大（4M block → 16× 已是过去式）。残余 2.5× 是协议开销，每一字节有效读在 NIC 上变成 2.5 字节流量：

| 层面 | 开销 | 说明 |
|------|------|------|
| FUSE 上下文切换 | 内核↔用户态 来回 | 每次 IO 至少 2 次切换 |
| TiKV 查元数据 | 读 chunk 前先查位置 | TCP 帧 + 往返延迟 |
| Ceph librados 协议 | 连接保活、认证、包头 | 每对象 ~几百字节框架 |
| EC 4+2 取片 | 256K 对象 = 4×64K shard × 4 OSD | 4 个独立 TCP 连接各带包头 |
| TCP/IP 头开销 | 每包 ~52 字节 | ~2% |

> 核心验证：rados bench 无 FUSE+TiKV → 145 ÷ 55.6 ≈ 2.6×，与 NIC 监控 2.5× 自洽。

### 7.6 多客户端结论

- **3 客户端聚合 55.6 MB/s，距目标 59 差 5.8%**
- **瓶颈 = 后端 EC 池 145 MB/s ÷ 残余读放大 2.5× = 58 MB/s 天花板**
- 4 客户端（node3 不通无法测）从趋势看无意义——已触后端天花板
- 要达标只剩两条路：**降残余放大 2.5→2.4×** 或 **抬后端 145→150+**（如升级 JuiceFS 版本、万兆网卡）
- 若验收口径接受 55.6，可认为已达标
